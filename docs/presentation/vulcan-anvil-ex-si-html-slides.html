01 / Why now
도입부
AI Coding Gap

AI는 코드를 빨리 만들지만,
SI 프로젝트는 “기록과 책임”으로 검수된다

대화창에 갇힌 요구사항

사용자 의도, 제약, 결정 근거가 에이전트 세션 안에 흩어진다.

코드와 산출물의 단절

요구사항·설계·테스트·증적이 코드 변경과 연결되지 않는다.

AI 결과의 책임 공백

누가 승인했고, 어떤 기준으로 통과했는지 설명하기 어렵다.

SI 현장에서는 “잘 돌아가는 코드”만으로는 부족하다. 요구사항, 산출물, 검수, 승인, 변경관리까지 남아야 한다.

05 / SI safety
도입부
Safe for SI

SI에 필요한 것은 자동화보다
통제 가능한 자동화다

사람 승인 지점

Gate 전환과 릴리즈 승인은 AI가 최종 확정하지 않고 사람의 승인 근거를 남긴다.

보안·기밀 경계

docs/ref-docs, 모델별 adapter, runner 실행 범위를 분리해 민감 정보 노출을 통제한다.

실패 경로 관리

반려, 변경요청, 백로그, 재검수 흐름을 문서와 명령으로 남긴다.

프레임워크의 목적은 “AI에게 맡기기”가 아니라, AI가 한 일을 SI 계약·감리·인수인계 문맥에서 설명 가능하게 만드는 것이다.

10 / Run-first
실체 구조
Runtime-first → Run-first

Claude/Codex 세션이 주인이 아니라
Run이 작업 계약이 된다

Runtime-first의 한계

세션 중심어느 채팅창에서 시작했는지가 전체 작업 흐름을 지배
작업 근거 흩어짐프롬프트, 판단, 결과가 Run 기록으로 남지 않음

Run-first의 전환

Run = 작업 계약범위, 입력문서, 수정 가능 경로, 검증 기준을 명시
Runner = 실행기Codex/Claude/Gemini는 같은 Run을 수행하는 worker
