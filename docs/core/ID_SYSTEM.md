# Vulcan-Anvil Ex ID 체계

> 상태: 초안 v0.1
> 목적: Vulcan-Anvil Ex 산출물을 단순하고 추적 가능하게 관리하기 위한 ID 체계를 정의한다.

## 1. 기본 원칙

Vulcan-Anvil Ex의 ID는 사람과 에이전트가 실제 개발 중에 쉽게 사용할 수 있을 만큼 단순해야 한다.

Core 원칙:

- 승인된 ID는 안정적으로 유지한다.
- 프로젝트별 코드표를 보지 않아도 대략적인 의미를 알 수 있어야 한다.
- 요구사항, 설계, 구현, 테스트, 승인까지 이어지는 추적성을 지원해야 한다.
- ID 안에 너무 많은 의미를 넣지 않는다. 상세 정보는 ID가 아니라 문서 메타데이터에 둔다.
- 프로젝트별 감리/고객 규칙은 확장으로 둘 수 있지만, Core ID 체계는 가볍게 유지한다.

Core ID 체계는 업무분류 자리수, 화면 계층 코드, 조직별 승인 단계처럼 특정 SI 프로젝트에 강하게 묶인 복잡한 문서 규칙을 그대로 복사하지 않는다.

## 2. ID 형식

기본 형식:

```text
PREFIX-001
PREFIX-002
PREFIX-003
```

예시:

```text
REQ-001
FUNC-001
SCR-001
PGM-001
UT-001
```

규칙:

- Prefix는 대문자를 사용한다.
- Prefix와 번호 사이에는 하이픈을 사용한다.
- 번호는 기본적으로 세 자리 숫자를 사용한다.
- 삭제된 ID는 재사용하지 않는다.
- 승인된 ID는 공식 변경 요청 없이 변경하지 않는다.
- 한 범주에서 999개를 넘을 가능성이 있는 프로젝트는 시작 시점부터 네 자리 숫자를 사용할 수 있다.

## 3. Core Prefix

| Prefix | 이름 | 의미 |
| --- | --- | --- |
| `DOC` | 문서 | 공식 프로젝트 문서 또는 템플릿 인스턴스 |
| `REQ` | 기능 요구사항 | 사용자가 기대하는 기능 요구사항 |
| `NREQ` | 비기능 요구사항 | 품질, 성능, 보안, 운영, 컴플라이언스, 제약 조건 |
| `AC` | 인수기준 | 요구사항 또는 기능을 검증할 수 있는 기준 |
| `FUNC` | 기능 | 요구사항에서 도출된 사용자/업무 기능 |
| `SCR` | 화면 | 화면, 페이지, 뷰, 다이얼로그, 팝업, 보고서 UI |
| `UI` | 화면테스트 | 화면 렌더링, 화면 이벤트, 반응형, 캡처 증적 테스트 |
| `PGM` | 프로그램 | API, 배치, 모듈, 서비스, 컴포넌트, 실행 단위 |
| `DB` | 데이터 객체 | 테이블, 컬렉션, 엔티티, 데이터 파일, 주요 데이터 객체 |
| `IF` | 인터페이스 | 시스템/모듈 간 내부 또는 외부 인터페이스 |
| `SEC` | 보안항목 | 보안 요구사항, 보안 설계 통제, 보안 검증 항목 |
| `UT` | 단위테스트 | 단위 또는 컴포넌트 수준 테스트 케이스 |
| `IT` | 통합테스트 | 여러 화면/프로그램/시스템을 연결해 검증하는 테스트 케이스 |
| `PT` | 성능테스트 | 성능, 부하, 스트레스, 용량 테스트 케이스 |
| `ISSUE` | 이슈 | 결함, 미해결 사항, 조치 항목 |
| `FIND` | QA 발견사항 | QA/리뷰 중 발견된 결함, 확인사항, G4 내부 수정 대상 |
| `RISK` | 위험 | 프로젝트, 기술, 보안, 일정, 품질 위험 |
| `CR` | 변경요청 | 요구사항, 범위, 설계, 베이스라인 변경 요청 |
| `DEC` | 의사결정 | 아키텍처, 제품, 프로세스, 범위 관련 의사결정 기록 |
| `RUN` | 에이전트 실행 | 에이전트에게 위임한 작업 단위, 실행 결과, 중단/완료 기록 |

## 4. 문서 ID

문서 ID는 다음 형식을 사용한다.

```text
DOC-{AREA}-{GATE}-{NNN}
```

예시:

```text
DOC-CORE-G1-001
DOC-CORE-G2-001
DOC-CORE-G3-001
DOC-PM-OPS-001
```

권장 영역 코드:

| 영역 | 의미 |
| --- | --- |
| `CORE` | 핵심 개발 산출물 |
| `PM` | 프로젝트 관리 산출물 |
| `QA` | 품질 또는 감리 산출물 |
| `SEC` | 보안 산출물 |
| `DATA` | 데이터 표준 또는 데이터 모델 산출물 |
| `DEV` | 개발 표준, 코드 구조, 빌드, 구현 규칙 산출물 |
| `OPS` | 운영, 배포, 인수인계 산출물 |
| `ADAPTER` | 에이전트 Adapter, 실행 규약, 모델별 연결 산출물 |

권장 Phase/Gate 코드:

| 코드 | 의미 |
| --- | --- |
| `P0` | Discovery / Phase 0 |
| `G1` | 요구사항 |
| `G2` | 설계 |
| `G3` | 테스트 계획 |
| `G4` | 검증 |
| `G5` | 승인 및 릴리즈 |
| `OPS` | 운영 |

## 5. 파일명 규칙

공식 문서 파일명의 기본 형식:

```text
{DOC-ID}_{document-name}_v{version}.{ext}
```

예시:

```text
DOC-CORE-G1-001_Requirements-Spec_v0.1.md
DOC-CORE-G2-001_Program-Spec_v0.9.md
DOC-QA-G4-001_Test-Result_v1.0.xlsx
```

규칙:

- 파일명에는 공백을 넣지 않는다.
- 이식성을 위해 파일명에는 영어 slug를 우선 사용한다.
- 사람이 읽는 한국어 문서명은 문서 내부 메타데이터에 둘 수 있다.
- 공식 산출물 파일명에는 문서 ID와 버전을 포함한다.

## 6. 버전 규칙

공식 산출물은 단순한 문서 버전 체계를 사용한다.

| 버전 | 의미 |
| --- | --- |
| `v0.1` | 최초 초안 |
| `v0.2` ~ `v0.8` | 내부 수정본 |
| `v0.9` | 검토 요청본 |
| `v1.0` | 승인된 베이스라인 |
| `v1.1` | 승인 후 경미한 변경 |
| `v2.0` | 승인 후 대규모 변경 |

규칙:

- 초안 문서는 `v0.1`부터 시작한다.
- 승인된 문서는 `v1.0`이 된다.
- 승인 이후 경미한 변경은 `0.1` 단위로 올린다.
- 대규모 변경은 주 버전을 올릴 수 있다.
- 회의록, 주간보고서, 일회성 작업 메모는 프로젝트에서 베이스라인으로 지정하지 않는 한 버전 관리를 강제하지 않는다.

## 7. 프로젝트별 외부 ID와의 관계

프로젝트에 이미 RFP ID, 감리 ID, 티켓 ID, 시스템 자동 생성 ID가 있을 수 있다.

이 ID들은 대체하지 않고 외부 참조로 보관한다.

```yaml
id: REQ-001
external_ids:
  rfp: SFR-07-012
  jira: PROJ-123
```

이렇게 하면 Ex Core의 추적성은 안정적으로 유지하면서, 감리 증적에 필요한 외부 ID도 함께 보존할 수 있다.
