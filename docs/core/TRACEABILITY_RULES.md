# Vulcan-Anvil Ex 추적성 규칙

> 상태: 초안 v0.1
> 목적: 요구사항, 설계 산출물, 보안항목, 테스트, 승인 증적이 어떻게 연결되어야 하는지 정의한다.

## 1. 핵심 규칙

의미 있는 모든 산출물은 다음 질문에 답할 수 있어야 한다.

> 이 산출물은 어떤 요구사항, 인수기준, 보안항목, 비기능 요구사항, 변경요청을 증명하거나 지원하는가?

테스트는 테스트 자체를 위해 존재하지 않는다. 모든 테스트는 다음 중 하나 이상을 검증해야 한다.

- `AC`
- `REQ`
- `NREQ`
- `SEC`
- `CR`

## 2. 최소 추적 경로

최소 성공 추적 경로는 다음과 같다.

```text
REQ -> AC -> FUNC -> SCR/PGM -> UT/IT -> Result
```

보안 민감 기능은 다음 경로를 가져야 한다.

```text
REQ/NREQ -> SEC -> SCR/PGM -> UT/IT -> Result
```

성능 민감 기능은 다음 경로를 가진다.

```text
NREQ -> PGM/IF/DB -> PT -> Result
```

## 3. 권장 산출물 연결

| 출발 ID | 반드시 연결 | 선택 연결 |
| --- | --- | --- |
| `REQ` | `AC`, `FUNC` | `NREQ`, `CR`, `DEC` |
| `NREQ` | `AC` 또는 검증 기준 | `SEC`, `PT`, 운영 문서 |
| `AC` | `REQ` 또는 `NREQ`, 테스트 ID | `SEC` |
| `FUNC` | `REQ`, `AC` | `SCR`, `PGM`, `DB`, `IF` |
| `SCR` | `FUNC` 또는 `REQ` | `SEC`, `PGM`, `UT`, `IT` |
| `PGM` | `FUNC` 또는 `REQ` | `SCR`, `DB`, `IF`, `SEC`, `UT`, `IT` |
| `DB` | `REQ`, `FUNC`, 또는 `PGM` | `SEC`, `PT` |
| `IF` | `REQ`, `FUNC`, 또는 `PGM` | `SEC`, `IT`, 외부 시스템 ID |
| `SEC` | `REQ`, `NREQ`, `SCR`, 또는 `PGM` | `UT`, `IT`, 보안 가이드 참조 |
| `UT` | `PGM`, `AC` 또는 `SEC` | 소스 파일 경로, 테스트 명령 |
| `IT` | `REQ` 또는 `AC` | `SCR`, `PGM`, `IF`, `DB` |
| `PT` | `NREQ` | `PGM`, `DB`, `IF`, 테스트 환경 |
| `CR` | 영향받는 ID | 의사결정, 승인, 릴리즈 노트 |

## 4. 인수기준과 단위테스트

인수기준은 기능 의도와 검증을 연결하는 다리다.

예시:

```text
REQ-001 회원가입 기능 제공
AC-001 정상 입력 시 계정이 생성된다
AC-002 중복 이메일이면 가입이 차단된다
AC-003 비밀번호 정책을 만족하지 않으면 오류를 표시한다

PGM-001 회원가입 API
UT-001 정상 입력 시 users 레코드가 생성된다 -> AC-001
UT-002 중복 이메일 입력 시 오류를 반환한다 -> AC-002
UT-003 약한 비밀번호 입력 시 오류를 반환한다 -> AC-003
```

규칙:

- 단위테스트는 프로그램 동작을 검증한다.
- 단위테스트가 필요한 이유는 `AC`, `SEC`, `NREQ` 중 하나에서 나와야 한다.
- 프레임워크 기본 동작만 증명하는 테스트를 필수 테스트로 만들지 않는다.
- 경계값, 예외, 권한, 보안 케이스가 별도 인수조건이나 위험을 나타낸다면 테스트도 분리한다.

## 5. 보안 추적성

보안항목은 화면명세, 프로그램명세, API 명세, DB 명세, 테스트 케이스에 포함될 수 있다.

규칙:

- `SEC` 항목은 구체적인 통제 또는 점검 내용을 설명해야 한다.
- `SEC` 항목은 영향을 받는 `SCR`, `PGM`, `DB`, `IF`와 연결하는 것이 좋다.
- 보안 민감 `SEC` 항목은 최소 하나 이상의 검증 산출물과 연결해야 한다.
- 보안항목은 외부 가이드를 참조할 수 있지만, 프로젝트 안에서 실제 구현 규칙을 별도로 정의해야 한다.

예시:

```text
SEC-001 비밀번호는 평문으로 저장하지 않는다.
적용 대상: PGM-001, DB-001
검증 대상: UT-004, IT-002
참조: 소프트웨어 개발보안 가이드 / 비밀번호 저장
```

## 6. 요구사항추적표 최소 컬럼

요구사항추적표는 최소한 다음 컬럼을 가진다.

| 컬럼 | 의미 |
| --- | --- |
| `REQ-ID` | 기능 요구사항 ID |
| `NREQ-ID` | 비기능 요구사항 ID. 해당 시 기재 |
| `AC-ID` | 인수기준 ID |
| `FUNC-ID` | 기능 ID |
| `SCR-ID` | 화면 ID |
| `PGM-ID` | 프로그램/API/배치/모듈 ID |
| `DB-ID` | 데이터 객체 ID |
| `IF-ID` | 인터페이스 ID |
| `SEC-ID` | 보안항목 ID |
| `UT-ID` | 단위테스트 ID |
| `IT-ID` | 통합테스트 ID |
| `PT-ID` | 성능테스트 ID |
| `상태` | 초안, 검토요청, 승인, 구현완료, 검증완료, 변경 |
| `증적` | 테스트 결과, 커밋, 파일 경로, 스크린샷, 보고서, 승인 기록 |

## 7. Gate별 기대 추적성

| Gate | 추적성 기대 수준 |
| --- | --- |
| Phase 0 | 의사결정과 가정을 메모 또는 `DEC` 항목으로 남긴다 |
| Gate 1 | 승인 범위의 `REQ`, `NREQ`, `AC`가 존재한다 |
| Gate 2 | 필요한 `FUNC`, `SCR`, `PGM`, `DB`, `IF`, `SEC`가 연결된다 |
| Gate 3 | `UT`, `IT`, `PT` 계획이 `AC`, `SEC`, `NREQ`와 연결된다 |
| Gate 4 | 테스트 결과와 리뷰 증적이 연결된다 |
| Gate 5 | 승인된 베이스라인과 릴리즈 증적이 연결된다 |

## 8. 추적성 결함 기준

다음 항목은 추적성 결함으로 본다.

- 인수기준이 없는 요구사항
- 검증 계획이 없는 인수기준
- 관련 기능 또는 요구사항이 없는 프로그램명세
- 영향받는 산출물과 연결되지 않은 보안항목
- 검증 계획이 없는 보안 민감 산출물
- 요구사항, 인수기준, 보안항목, 비기능 요구사항 중 어느 것과도 연결되지 않은 테스트 케이스
- 영향받는 ID가 없는 승인된 변경요청
