# 보안 기준선

> 상태: 초안 v0.1
> 목적: 요구사항, 설계, 구현, 테스트 단계에서 반복적으로 참조할 공통 보안 기준을 정의한다.

본 문서는 공통 기준선이다. 프로젝트별 실제 구현 값, 입력 정책, 오류 메시지, 적용 위치, 테스트 ID는 Gate 2의 보안가이드(`docs/artifacts/02-design/security/DOC-SEC-G2-001_Security-Guide_v0.1.md`)에 확정한다.

## 1. 기본 원칙

에이전트는 보안 설계가 필요할 때마다 원문 PDF 전체를 매번 읽지 않는다.

대신 다음 순서로 판단한다.

1. 본 문서에서 기능 유형별 필수 보안항목을 식별한다.
2. 식별된 항목을 `SEC-ID`, `KISA/SR`, `OWASP`, `CWE`, `TEST-ID`와 연결한다.
3. 세부 판단이 필요할 때만 `docs/seed-docs/reference-standards/`의 원문을 확인한다.
4. 원문과 본 문서가 충돌하면 원문을 우선하고, 본 문서 갱신 후보를 `FIND` 또는 `CR`로 남긴다.

본 문서는 다음 세 기준을 기본 축으로 사용한다.

| 기준 | 역할 | Ex 사용 방식 |
| --- | --- | --- |
| 행안부/KISA `SR-ID` | 국내 공공 개발보안 및 감리 대응 기준 | 요구사항, 설계, 테스트의 1차 근거 |
| OWASP Top 10 | 웹 애플리케이션 위험 분류 | 위험 유형 및 누락 체크 |
| CWE | 코드 레벨 보안약점 분류 | 구현/리뷰/진단 시 약점 ID 연결 |

## 2. 참조 표준

| 참조 코드 | 문서/표준 | 위치 또는 URL | 용도 |
| --- | --- | --- | --- |
| `KISA-SD-2021` | 소프트웨어 개발보안 가이드(2021.12.29) | `docs/seed-docs/reference-standards/` | 설계/구현 보안 기준 |
| `KISA-SWVD-2021` | 소프트웨어 보안약점 진단가이드(2021) | `docs/seed-docs/reference-standards/` | 감리/진단/검증 기준 |
| `OWASP-T10-2021` | OWASP Top 10:2021 | `https://owasp.org/Top10/2021/` | 웹 위험 분류 |
| `CWE` | Common Weakness Enumeration | `https://cwe.mitre.org/` | 코드 약점 분류 |

## 3. 보안항목 작성 규칙

`SEC-ID`는 내부 추적 ID이며, 외부 기준 ID를 반드시 함께 가진다.

권장 형식:

```yaml
security_id: SEC-001
name: 인증 실패 처리
description: 인증 실패 시 세션을 생성하지 않고 계정 존재 여부를 노출하지 않는다.
applies_to:
  - REQ-001-02
  - PGM-001
  - SCR-001
reference_standards:
  kisa_sr:
    - KISA-SD-2021 SR2-1
    - KISA-SD-2021 SR2-2
  owasp:
    - OWASP-T10-2021 A07
  cwe:
    - CWE-287
    - CWE-209
verification:
  - UT-002
  - IT-001
```

필수 규칙:

- 인증, 권한, 비밀번호, 세션, 개인정보, 파일, 외부 입력, 외부 연계가 있으면 `SEC-ID` 후보를 만든다.
- `SEC-ID`는 최소 하나 이상의 `REQ` 또는 `NREQ`와 연결한다.
- `SEC-ID`는 최소 하나 이상의 `PGM`, `SCR`, `DB`, `IF` 중 하나와 연결한다.
- `SEC-ID`는 최소 하나 이상의 테스트 또는 리뷰 증적과 연결한다.
- 공공/SI 프로젝트에서는 `KISA/SR` 매핑을 먼저 작성하고, OWASP/CWE는 보조 분류로 추가한다.

## 4. 기능 유형별 필수 보안항목

### 4.1 인증/로그인

로그인, 사용자 식별, 비밀번호 검증, 세션 발급 기능이 있으면 다음 항목을 검토한다.

| SEC 후보 | 보안항목 | KISA/SR | OWASP | CWE | 요구사항/설계 반영 | 테스트 예 |
| --- | --- | --- | --- | --- | --- | --- |
| `SEC-AUTH-001` | 인증 대상 및 방식 정의 | `SR2-1` | `A07` | `CWE-287` | 로그인 대상, 인증 방식, 우회 방지 조건 | 정상/실패 로그인 |
| `SEC-AUTH-002` | 인증 실패 처리 및 계정 존재 여부 미노출 | `SR2-1`, `SR3-1` | `A07`, `A05` | `CWE-209`, `CWE-287` | 실패 메시지, 실패 응답, 세션 미생성 | 실패 로그인 |
| `SEC-AUTH-003` | 로그인 시도 제한 | `SR2-2` | `A07` | `CWE-307` | 실패 횟수 제한, 잠금, 감사 로그 | 반복 실패 |
| `SEC-AUTH-004` | 비밀번호 정책 | `SR2-3` | `A07` | `CWE-521` | 길이, 복잡도, 금지 패턴, 변경 정책 | 약한 비밀번호 거부 |
| `SEC-AUTH-005` | 비밀번호 저장 보호 | `SR2-3`, `SR2-6`, `SR2-7` | `A02`, `A07` | `CWE-256`, `CWE-916` | 평문 저장 금지, salt/반복 기반 해시 | 저장값 검증 |
| `SEC-AUTH-006` | 세션 토큰 보호 | `SR4-1`, `SR2-8` | `A01`, `A07` | `CWE-384`, `CWE-613` | 세션 생성, 만료, 재사용 방지, 전송 보호 | 세션 접근 |

| `SEC-AC-002` | 객체 단위 권한 검증 | `SR2-4` | `A01` | `CWE-639` | 게시글/첨부/사용자별 소유권 | 타 사용자 자원 접근 |

## 5. Gate별 적용 규칙

| Gate | 필수 수행 |
| --- | --- |
| P0 | 보안 민감 도메인, 개인정보, 외부연계, 인증/권한 여부를 질문으로 남긴다. |
| G1 | 기능 유형별 필수 보안항목을 `NREQ`, `SEC`, `AC` 후보로 식별한다. |
| G2 | 보안가이드에서 `KISA/SR` 근거, 프로젝트 상세스펙, OWASP/CWE 보조 분류를 확정하고 프로그램 설계, 화면설계, DB명세에 반영한다. |
| G3 | 모든 `SEC-ID`의 구체 규격에 보안 테스트 또는 리뷰 증적 기준을 연결한다. |
| G4 | 구현 코드가 보안 기준을 만족하는지 확인하고, 미흡하면 `FIND` 또는 `CR`로 분류한다. |
| G5 | 미해결 보안 Finding, 보안 테스트 미실행, CR 승격 대상을 확인한다. |

## 6. 보안 요구사항 식별 체크리스트

요구사항 persona와 security-review persona는 다음 질문에 하나라도 Yes가 있으면 `SEC-ID` 후보를 작성한다.

| 질문 | Yes일 때 후보 |
| --- | --- |
| 로그인이 있는가 | `SEC-AUTH-001` ~ `SEC-AUTH-006` |
| 비밀번호를 입력, 변경, 저장하는가 | `SEC-AUTH-004`, `SEC-AUTH-005` |
| 로그인 실패를 처리하는가 | `SEC-AUTH-002`, `SEC-AUTH-003` |
| 사용자별 권한이 다른가 | `SEC-AC-001`, `SEC-AC-002` |
| 사용자 입력을 DB 조회나 저장에 사용하는가 | `SEC-IN-001`, `SEC-IN-002` |
| HTML 화면에 사용자 입력을 출력하는가 | `SEC-IN-003` |
| 파일 업로드/다운로드가 있는가 | `SEC-FILE-001` ~ `SEC-FILE-003` |
| 개인정보나 인증정보를 저장/전송하는가 | `SEC-DATA-001`, `SEC-DATA-002` |
| 외부 URL이나 외부 API를 호출하는가 | `SEC-API-001` ~ `SEC-API-003` |
| 운영 설정, 기본 계정, 디버그 모드가 있는가 | `SEC-CONF-001`, `SEC-CONF-002` |

## 9. 미해결 또는 주의사항

- OWASP Top 10과 CWE는 계속 개정될 수 있으므로 프로젝트 시작 시 기준 버전을 확정한다.
- KISA `SR-ID`는 설계단계 보안설계 기준이다. 구현단계 세부 보안약점은 `KISA-SWVD-2021`과 언어별 시큐어코딩 가이드를 추가 확인한다.
- 비밀번호 저장 보호는 단순 해시가 아니라 salt와 충분한 반복/전용 알고리즘 적용 여부를 검토해야 한다.
- 본 문서는 기준선이며, 프로젝트 도메인 특화 보안 요구사항은 별도 `SEC-ID`로 추가한다.
