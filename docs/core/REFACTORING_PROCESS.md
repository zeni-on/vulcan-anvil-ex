# 리팩토링 프로세스

> 상태: 초안 v0.1
> 목적: 리팩토링을 추적 가능한 개선 작업으로 관리하고, 설계 문서와 테스트/증적에 미치는 영향을 판단하는 기준을 정의한다.

## 1. 핵심 원칙

리팩토링은 기능을 새로 만드는 작업이 아니지만, 문서와 설계에 영향을 줄 수 있다.

Vulcan-Anvil Ex에서는 리팩토링을 다음처럼 다룬다.

```text
기능 변경 없는 내부 개선 = DEBT 또는 FIND
설계 문서에 영향이 있는 구조 변경 = 문서 갱신 또는 CR
외부 계약을 바꾸는 변경 = CR 필수
```

에이전트가 코딩하고, Vulcan-Anvil이 그 일을 설명 가능하게 만든다.

따라서 리팩토링은 "그냥 정리했다"로 끝내지 않는다. 사유, 영향 범위, 변경 파일, 영향 문서, 재실행한 테스트, 결과를 Run 또는 Backlog/CR에 남긴다.

## 2. 리팩토링 분류

| 유형 | 의미 | 기본 처리 |
| --- | --- | --- |
| 내부 품질 개선 | 메서드 분리, 중복 제거, 네이밍 정리, 작은 구조 정돈 | `DEBT` 또는 `FIND` + Run |
| 개발표준 정렬 | 계층 분리, 패키지 구조 정리, 메시지 리소스 분리, 테스트 구조 정리 | `DEBT` 또는 `FIND`, 영향 문서 갱신 |
| 설계 영향 리팩토링 | 프로그램 구조, API 응답, DB 접근 방식, 인증/인가 구조 변경 | 영향도 판단 후 `CR` 가능 |
| 외부 계약 변경 | API/DB/UI/권한/보안/성능 기준 변경 | `CR` 필수 |
| 기술스택 변경 | 프레임워크, ORM, 상태관리, 인증 방식 변경 | `CR` + `ADR` + Gate 2 재진입 |

## 3. 문서 영향 판단

| 리팩토링 예 | 문서 영향 |
| --- | --- |
| 메서드 이름 변경, 중복 코드 제거 | 보통 설계문서 영향 없음. Run/커밋/테스트 결과 기록 |
| 패키지 구조 변경 | 개발표준정의서 영향 |
| Controller-Service-Repository 계층 분리 | 프로그램 설계서, 개발표준정의서 영향 |
| Entity/DTO 분리 | 프로그램 설계서, API정의서, DB명세서 영향 가능 |
| 인증 필터 구조 변경 | SW Architecture, 보안가이드, 프로그램 설계서 영향 |
| API 응답 구조 변경 | API정의서, 화면설계서, 테스트케이스 영향 |
| 화면 상태관리 구조 변경 | 화면설계서, 프로그램 설계서, 테스트케이스 영향 |
| DB 조회 방식 변경 | 프로그램 설계서, DB명세서, 성능/보안 테스트 영향 가능 |

## 4. 처리 흐름

```text
리팩토링 필요 발견
→ 사유와 관련 ID 확인
→ 기능/외부 계약 변경 여부 판단
→ 문서 영향 여부 판단
   → 영향 없음: DEBT/FIND + Refactoring Run
   → 설계 문서 영향 있음: 관련 문서 갱신 + Run
   → 요구/API/DB/UI/보안 기준 변경 있음: CR 승격
→ 테스트/검증 명령 실행
→ Run, Test Result, Traceability, Backlog/CR 상태 갱신
```

## 5. DEBT로 처리 가능한 경우

다음 조건을 모두 만족하면 보통 `DEBT` 또는 `FIND`로 처리할 수 있다.

| 조건 | 기준 |
| --- | --- |
| 기능 의미 변경 없음 | `REQ`, `AC`, `NREQ` 의미가 바뀌지 않는다. |
| 외부 계약 변경 없음 | API, DB, UI, 권한, 보안 정책이 바뀌지 않는다. |
| 설계 기준 유지 | SW Architecture, 개발표준, 프로그램 설계의 기본 구조를 유지한다. |
| 영향 범위 한정 | 변경 파일과 관련 ID를 명확히 열거할 수 있다. |
| 회귀 검증 가능 | 관련 테스트 또는 필수 검증 명령을 실행할 수 있다. |

예:

- 중복 로직을 private method로 추출
- 테스트 helper 정리
- 메시지 문자열을 기존 메시지 리소스로 이동
- package 위치를 개발표준에 맞게 조정하되 외부 계약 변화 없음
- Controller에서 반복되던 validation 변환 코드를 공통 유틸로 분리

## 6. CR로 승격해야 하는 경우

다음 중 하나라도 해당하면 `CR`로 승격한다.

| 기준 | 이유 |
| --- | --- |
| API 요청/응답 구조 변경 | 외부 계약 변경 |
| DB 테이블/컬럼/인덱스 변경 | 데이터 설계 변경 |
| 화면 흐름, 상태, 입력 항목 변경 | 화면설계와 테스트 기준 변경 |
| 인증/인가/세션/토큰 방식 변경 | 보안 기준 변경 |
| 기술스택 또는 주요 라이브러리 교체 | 아키텍처/개발표준 변경 |
| 기존 테스트 기준 삭제 또는 완화 | 검증 기준 변경 |
| 성능/가용성/운영 기준 변경 | 비기능 요구사항 변경 가능 |

CR로 승격되면 `CHANGE_CONTROL_PROCESS.md`에 따라 필요한 Gate를 다시 진행한다.

## 7. 리팩토링 Run 필수 기록

리팩토링 Run은 최소한 다음을 남긴다.

| 항목 | 내용 |
| --- | --- |
| 리팩토링 사유 | 왜 필요한가 |
| 기능 변경 여부 | 없음 / 있음 |
| 외부 계약 변경 여부 | 없음 / 있음 |
| 영향받는 문서 | 없음 또는 문서 ID |
| 영향받는 코드 | 파일/모듈 |
| 영향받는 ID | `REQ/AC/FUNC/SCR/PGM/API/DB/SEC/UT/IT` |
| 재실행한 검증 명령 | 개발표준정의서의 필수 검증 명령 또는 영향 범위 명령 |
| 결과 | Pass / Fail / Not Run / Skipped |
| 후속 조치 | 없음 / FIND / CR / DEBT |

## 8. 테스트 기준

리팩토링 후에는 범위에 맞는 검증을 실행한다.

| 변경 범위 | 최소 검증 |
| --- | --- |
| 작은 내부 정리 | 관련 단위테스트 |
| 공통 모듈 변경 | 전체 단위테스트 |
| API/Service 계층 변경 | 단위테스트 + 관련 통합/API 테스트 |
| 화면 구조 변경 | 컴포넌트 테스트 또는 E2E/UI 증적 |
| 보안 구조 변경 | 인증/인가/세션 보안 테스트 |
| DB 접근 변경 | Repository/통합 테스트, 주요 조회 성능 확인 |

자동화된 단위테스트가 있으면 원칙적으로 전체 실행한다. 통합/API/UI 테스트는 영향받는 흐름 중심으로 실행하되, 릴리즈 전에는 전체 회귀 범위를 다시 확인한다.

## 9. 추적성 기준

리팩토링은 새 기능이 아니어도 추적성이 필요하다.

| 상황 | 추적 방식 |
| --- | --- |
| 요구사항 의미 변경 없음 | 관련 `REQ/PGM/UT`와 Run을 연결 |
| 개발표준 위반 수정 | `DOC-DEV-G2-001`, 관련 파일, 테스트 결과 연결 |
| 보안 구조 정리 | `SEC-ID`, 보안가이드, 테스트 결과 연결 |
| API/DB/UI 계약 변경 | `CR-ID`와 관련 산출물 전체 연결 |
| 기술부채 처리 | Backlog `DEBT`, Run, 커밋 연결 |

## 10. 완료 조건

| 항목 | 기준 |
| --- | --- |
| 사유 | 리팩토링 목적이 설명되어 있다. |
| 영향도 | 문서, 코드, 테스트, 외부 계약 영향 여부가 기록되어 있다. |
| 문서 갱신 | 영향받는 설계/개발표준/API/DB/화면/보안 문서가 갱신되었다. |
| 테스트 | 필요한 검증 명령이 실행되고 결과가 기록되었다. |
| 추적 | 관련 ID, Run, 커밋, Backlog/CR 상태가 연결되어 있다. |
| 미해결 | 남은 위험이나 후속 작업이 Backlog/ISSUE/CR로 남아 있다. |
