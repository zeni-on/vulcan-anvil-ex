# Vulcan-Anvil Ex 변경관리 프로세스

> 상태: 초안 v0.1
> 목적: 프로젝트 기간 중 발생하는 수정, 결함, 요구 추가, 범위 변경을 QA 내부 수정 루프와 공식 변경요청 루프로 분리해 관리한다.

## 1. 핵심 원칙

변경 관리는 다음 원칙을 따른다.

```text
설계된 것을 제대로 만들기 위한 수정 = Finding/Defect
설계 자체를 바꾸는 수정 = Change Request
```

따라서 모든 발견사항을 곧바로 공식 변경요청으로 올리지 않는다. QA 단계에서 기존 요구사항과 설계 범위 안의 결함을 발견한 경우에는 G4 내부의 QA Fix Loop로 빠르게 수정하고 재검증한다.

반대로 요구사항, 인수조건, 설계, 보안 기준, 일정, 범위가 바뀌는 경우에는 반드시 `CR`로 승격한다.

## 2. 용어 구분

| 유형 | ID | 의미 | 대표 처리 |
| --- | --- | --- | --- |
| Finding | `FIND-001` | QA/리뷰 중 발견한 결함, 누락, 확인사항 | G4 QA Fix Loop |
| Issue | `ISSUE-001` | 아직 원인/처리 방향이 확정되지 않은 미해결 사항 | 분석 후 FIND/CR/RISK로 분류 |
| Change Request | `CR-001` | 요구사항, 범위, 설계, 베이스라인 변경 요청 | 영향도 분석 후 승인/반려/보류 |
| Backlog | `BL-001` | 승인됐지만 지금 처리하지 않는 작업 대기열 | 우선순위에 따라 Run 생성 |
| Run | `RUN-001` | 에이전트에게 위임한 실제 작업 단위 | 실행 결과와 검증 기록 |

## 3. 전체 흐름

```text
요청, 결함, 리뷰 의견, QA 실패 발생
→ ISSUE 또는 FIND 후보 등록
→ 기존 설계 안에서 고치는 문제인지 판단
   → Yes: FIND 등록 후 G4 QA Fix Loop 수행
   → No: CR 등록
→ CR 영향도 분석
→ 승인/반려/보류
→ 승인 시 Backlog 또는 즉시 Run 생성
→ 필요한 Gate로 증분 재진입
→ 산출물, 코드, 테스트, 증적, 추적표 갱신
```

## 4. G4 QA Fix Loop

G4 QA Fix Loop는 QA 단계에서 발견한 결함을 같은 Gate 안에서 재귀적으로 수정하는 흐름이다.

```text
G4 Review Run
→ FIND 등록
→ QA 내부 수정 가능 여부 판단
→ QA Fix Run
→ 테스트 재실행
→ 증적 갱신
→ 추적표 상태 갱신
→ Review Run 재실행
→ Pass면 G5 후보
```

다음 조건을 모두 만족하면 QA Fix Loop로 처리할 수 있다.

| 판단 기준 | 설명 |
| --- | --- |
| 요구사항 의미 변경 없음 | `REQ`, `NREQ`의 의미가 바뀌지 않는다. |
| 인수조건 변경 없음 | `AC` 추가, 삭제, 의미 변경이 없다. |
| 설계 변경 없음 | `FUNC`, `SCR`, `PGM`, `DB`, `SEC`의 구조나 정책이 바뀌지 않는다. |
| 보안 기준 완화 없음 | 기존 보안 기준을 낮추지 않는다. |
| 테스트 기준 보강 수준 | 기존 기준을 증명하기 위한 테스트 추가 또는 실패 수정이다. |
| 영향 범위 한정 | 관련 ID와 변경 파일을 명확히 열거할 수 있다. |

QA Fix Loop 예시:

| 예 | 처리 |
| --- | --- |
| 기존 AC를 만족하지 못하는 코드 오류 | FIND 등록 후 G4 수정 |
| 기존 SEC 구현 누락 | FIND 등록 후 G4 수정 |
| 테스트 실패 원인이 구현 결함임 | FIND 등록 후 G4 수정 |
| 화면 문구 오탈자 | FIND 등록 후 G4 수정 |
| 증적 캡처 누락 | FIND 등록 후 Evidence Run |

## 5. CR 승격 기준

다음 중 하나라도 해당하면 `CR`로 승격한다.

| 승격 기준 | 재진입 Gate |
| --- | --- |
| 요구사항 추가, 삭제, 의미 변경 | G1 |
| 비기능 요구사항 또는 보안 요구사항 변경 | G1 |
| 인수조건 추가, 삭제, 의미 변경 | G1 또는 G3 |
| 기능, 화면, 프로그램, DB, 인터페이스 설계 변경 | G2 |
| 테스트 기준 자체 변경 | G3 |
| 승인된 범위, 일정, 우선순위 변경 | G1 또는 G5 |
| 보안 기준 완화 또는 예외 승인 필요 | G1 |
| 고객 요청에 의한 신규 기능 | G1 |

CR 승격 예시:

| 예 | 처리 |
| --- | --- |
| 게시글 첨부파일 추가 | CR 등록, G1 재진입 |
| 로그인 실패 정책 변경 | CR 등록, G1 또는 G2 재진입 |
| DB 컬럼 추가 | CR 등록, G2 재진입 |
| 화면 항목 추가 | CR 등록, G2 재진입 |
| 기존 AC를 더 엄격하게 변경 | CR 등록, G1/G3 재진입 |

## 6. 증분 Gate 재진입

CR이 승인되면 전체 프로젝트를 다시 진행하지 않고 영향받는 ID 범위만 재진입한다.

| 변경 영향 | 최소 재진입 Gate | 필수 갱신 |
| --- | --- | --- |
| `REQ`, `NREQ`, `AC` | G1 | 요구사항정의서, 추적표 |
| `FUNC`, `SCR`, `PGM`, `DB`, `SEC` | G2 | 설계 산출물, 추적표 |
| `UT`, `IT`, `PT`, `UI` | G3 | 테스트케이스, 추적표 |
| 구현/증적 | G4 | 코드, 테스트 결과, 화면/로그 증적 |
| 승인/릴리즈 | G5 | 승인본, 릴리즈 기록 |

증분 재진입은 반드시 scope를 가진다.

```text
scope: REQ-005, AC-008, FUNC-005, PGM-005, SEC-002, UT-008
```

## 7. Backlog와의 관계

Backlog는 CR 자체가 아니며, Gate 밖에 있는 단순 TODO도 아니다. Backlog는 Phase 0, FIND, CR, ISSUE, 기술부채를 다음 Run 또는 재진입 Gate로 연결하는 대기열이다.

| 상황 | 처리 |
| --- | --- |
| Phase 0 아이디어가 아직 요구사항으로 확정되지 않음 | `IDEA`로 Backlog 등록 후 Gate 1 후보로 정리 |
| CR 승인, 즉시 반영 | 바로 Run 생성 |
| CR 승인, 다음 배치 반영 | `CR`로 Backlog 등록 |
| CR 반려 | CR 상태를 Rejected로 기록 |
| CR 보류 | CR 상태를 Deferred로 기록하고 필요하면 `ISSUE` 또는 Backlog에 보류 사유 기록 |
| FIND가 다음 배치로 밀림 | Minor 이하만 `FIND`로 Backlog 등록 가능 |
| 판단이 불명확함 | `ISSUE`로 Backlog 등록 후 사용자 의사결정 요청 |
| 기술부채 또는 운영 개선 | `DEBT`로 Backlog 등록 |

Backlog 항목은 `유형`, `관련 ID`, `재진입 Gate`, `관련 Run`, `우선순위`를 함께 가져야 한다. G4에서 Blocker 또는 Major 결함은 Backlog로 넘기지 않는다. 현재 Gate 안에서 해결하거나 CR로 승격해야 한다.

## 8. 에이전트 판단 규칙

에이전트는 결함이나 변경 후보를 발견하면 다음 순서로 판단한다.

1. 관련 ID를 찾는다.
2. 기존 문서와 코드가 충돌하는지 확인한다.
3. 기존 설계를 만족시키는 수정이면 `FIND`로 처리한다.
4. 기존 설계를 바꿔야 하면 `CR`로 승격한다.
5. 판단이 불명확하면 `ISSUE`로 남기고 사용자 확인을 요청한다.
6. `FIND`, `CR`, `ISSUE` 모두 관련 `REQ/AC/FUNC/SCR/PGM/DB/SEC/UT/IT`를 기록한다.

에이전트는 `CR` 승인 없이 다음을 수행하지 않는다.

- 승인된 요구사항 의미 변경
- 보안 기준 완화
- DB 구조 변경
- 화면/업무 흐름 변경
- 테스트 기준 삭제 또는 완화
- 범위 밖 파일 대규모 수정

## 9. 완료 조건

### FIND 완료 조건

| 항목 | 기준 |
| --- | --- |
| 원인 | 기존 설계 내 결함임을 설명 |
| 수정 | 관련 코드/문서/테스트 수정 완료 |
| 검증 | 관련 테스트 재실행 결과 기록 |
| 증적 | 필요한 화면/로그/결과 증적 갱신 |
| 추적 | 관련 ID와 Run ID 연결 |

### CR 완료 조건

| 항목 | 기준 |
| --- | --- |
| 요청 | 변경 요청 내용과 출처 기록 |
| 영향도 | 영향받는 ID, 문서, 코드, 테스트 기록 |
| 승인 | Approved/Rejected/Deferred 상태 기록 |
| 재진입 | 필요한 Gate와 scope 기록 |
| 반영 | 산출물, 코드, 테스트, 증적 갱신 |
| 추적 | 추적표와 변경이력에 CR 반영 |

## 10. 최소 예시

### FIND 예시

```text
FIND-001: 비로그인 사용자가 게시글 작성 API 호출 시 401이 아닌 500 발생
관련 ID: REQ-005, AC-008, PGM-005, SEC-002, UT-008
처리 방식: G4 QA Fix Run
CR 필요 여부: No
사유: 기존 요구사항/설계 범위 내 결함 수정
```

### CR 예시

```text
CR-001: 게시글 첨부파일 추가 요청
관련 ID: 신규 REQ 후보, FUNC-005, SCR-005, PGM-005, DB-002
처리 방식: G1 증분 재진입
CR 필요 여부: Yes
사유: 신규 기능과 DB/화면/프로그램 설계 변경 필요
```
