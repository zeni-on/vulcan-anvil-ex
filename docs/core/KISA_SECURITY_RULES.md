## 2. 적용 방식

- `KISA 제안/근거`는 원문에서 직접 확인한 설계항목, 보안대책, 설계 시 고려사항을 짧게 요약한 것이다.
- `설계 시 확인할 규격`은 기능명세서, 화면설계서, 프로그램명세서, DB명세서에서 확인해야 할 체크 항목이다.
- `프로젝트 보안가이드에서 확정해야 할 값`은 프로젝트별로 반드시 값을 박아야 하는 구현 규칙이다.
- `주의: 프로젝트 정책으로 별도 결정해야 하는 값`은 KISA 원문에 없거나, 원문이 방향만 제시한 값이다. 예를 들어 비밀번호 해시 라이브러리의 입력 길이 제한 같은 구현 정책은 KISA 근거로 쓰지 않는다.

## 4. SR1-1 DBMS 조회 및 결과 검증

### KISA 제안/근거

- DBMS 조회 시 SQL 안의 입력값과 조회결과에 대한 유효성 검증방법 및 유효하지 않은 값의 처리방법을 설계해야 한다. (p.47, p.343)
- DB 연결 계정은 최소권한으로 설정해야 한다. 침해가 발생하더라도 해당 애플리케이션이 사용하는 데이터의 읽기, 쓰기, 삭제, 업데이트 권한으로 제한한다. (p.47-48)
- 외부 입력값이 SQL 구조를 바꾸지 않도록 동적 SQL 생성을 피하고, ORM 또는 안전한 정적 쿼리 구조를 사용하도록 개발환경과 코딩 규칙을 정한다. (p.48)
- 동적 SQL이 불가피하면 클라이언트와 서버 양측에서 입력값 검증을 수행하고, 필터, 인터셉터, Validator 컴포넌트를 공통으로 적용한다. (p.48-49)

### 설계 시 확인할 규격

- DB 접근 계정의 권한 범위가 기능별 최소권한인지 확인한다.
- 사용자 입력을 SQL 조건, 정렬, 페이지 크기, 검색어, 식별자에 사용하는 모든 위치를 식별한다.
- 정적 쿼리, 파라미터 바인딩, ORM 안전 API 사용 여부를 프로그램명세서에 적는다.
- 입력값 검증 실패 시 응답 코드, 사용자 메시지, 로그 기록 여부를 정한다.
- DB 조회결과를 다시 화면/API 응답으로 내보낼 때 `SR1-5`, `SR2-7`, `SR3-1`과 함께 검토한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- DB 계정명 규칙과 권한 범위.
- SQL 작성 금지 패턴과 허용 패턴.
- ORM/쿼리 빌더/바인딩 API 이름.
- 입력값별 허용 문자, 길이, 형식, 허용 목록.
- 검증 실패 시 공통 오류 코드, HTTP 상태, 로그 레벨.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 특정 ORM 또는 DB 드라이버 선택은 KISA 근거가 아니라 프로젝트 기술스택 결정이다.
- 검색어 최소/최대 길이, 페이지 크기, 정렬 컬럼 허용 목록은 업무와 성능 요구에 맞춰 별도 확정한다.
- SQL injection 방어를 정규식 필터만으로 완료했다고 판단하지 않는다. 구조적으로 SQL이 바뀌지 않는 API 사용을 우선한다.

## 5. SR1-5 웹 서비스 요청 및 결과 검증

### KISA 제안/근거

- 웹 서비스 요청과 응답결과에 대해 유효성 검증방법과 유효하지 않은 값의 처리방법을 설계해야 한다. (p.62, p.344)
- 사용자 입력값을 동적으로 생성되는 응답페이지에 사용할 경우 XSS 필터링을 수행한 뒤 사용해야 한다. (p.62-64)
- DB 조회결과를 응답페이지에 사용할 경우 HTML 인코딩 또는 XSS 필터링을 수행한 뒤 사용해야 한다. (p.62, p.64-65)
- DB에서 읽어온 값도 외부 입력값의 범위에 포함하여 응답페이지 출력 전 검증해야 한다. (p.65)
- View 컴포넌트에서 출력값에 HTML 인코딩 또는 XSS 필터를 적용하도록 시큐어코딩 규칙을 정의한다. (p.64)

### 설계 시 확인할 규격

- 사용자 입력이 HTML, 속성, URL, JavaScript, CSS, JSON 등 어느 출력 컨텍스트로 나가는지 분리한다.
- 저장형 XSS 방지를 위해 DB 저장값을 화면에 표시하는 모든 위치를 식별한다.
- 화면 컴포넌트, 템플릿 엔진, 프론트엔드 프레임워크의 기본 escaping 동작과 예외 API를 확인한다.
- HTML 허용 입력이 필요한 경우 sanitize 정책과 허용 태그/속성을 별도 정의한다.
- API 응답을 HTML로 렌더링하는 클라이언트 코드의 안전한 렌더링 규칙을 정한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 출력 컨텍스트별 인코딩 함수 또는 공통 컴포넌트.
- sanitize 라이브러리와 허용 태그/속성 목록.
- raw HTML 렌더링 금지 또는 예외 승인 절차.
- XSS 검증 테스트 payload와 기대 결과.
- 화면/컴포넌트별 사용자 입력 출력 위치.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 특정 XSS 라이브러리 선택은 프로젝트 기술스택과 유지보수성을 기준으로 확정한다.
- 프레임워크 기본 escaping이 있는 경우에도 `dangerouslySetInnerHTML`, `innerHTML`, template bypass 같은 예외 API 사용 정책을 별도로 둔다.
- HTML 인코딩과 입력값 삭제는 목적이 다르다. 저장 전 필터링만으로 모든 출력 컨텍스트가 안전해진다고 보지 않는다.

## 6. SR2-1 인증 대상 및 방식

### KISA 제안/근거

- 중요정보와 중요기능의 특성에 따라 인증방식을 정의하고, 정의된 인증방식을 우회하지 못하게 설계해야 한다. (p.88, p.346)
- 중요기능이나 리소스에는 인증 후 사용 정책을 적용해야 한다. (p.88-89)
- 인증정보는 서버 측에 저장하고, 인증이 필요한 기능이나 리소스 접근 시 서버에서 인증 여부를 확인해야 한다. (p.89)
- 중요기능은 2단계 인증을 고려한다. 2단계 인증은 지식, 소유, 생체 기반 인증 중 2개 이상의 인증기법을 사용하는 방식으로 설명된다. (p.89-90)
- 중앙집중식 인증이 적용되도록 인증 기능을 제공하는 프레임워크 또는 라이브러리 활용을 고려한다. (p.89-90)

### 설계 시 확인할 규격

- 인증이 필요한 중요기능과 중요 리소스를 목록화한다.
- 익명 사용자, 인증 사용자, 관리자, 시스템 계정 등 호출자 유형을 구분한다.
- 인증 검사가 라우터, 미들웨어, 서비스 계층 중 어디에서 강제되는지 정한다.
- 인증 우회 가능한 직접 URL, API, 파일 다운로드, 관리자 페이지 진입 경로를 점검한다.
- 고위험 기능에 추가인증 또는 재인증이 필요한지 판단한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 인증 방식: 세션, 토큰, SSO, OAuth/OIDC, 인증서, 2FA 등.
- 인증 적용 대상 URL/API/기능 목록.
- 인증 실패 시 응답 코드와 사용자 메시지.
- 인증 상태 저장 위치와 검증 함수.
- 추가인증 적용 기준과 유효시간.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 2FA 적용 여부와 방식은 업무 위험도, 사용자 환경, 운영 비용을 기준으로 확정한다.
- JWT, 세션 쿠키, 서버 세션 저장소 같은 구현 방식은 KISA가 특정하지 않으므로 프로젝트에서 정한다.
- 클라이언트의 화면 숨김이나 버튼 비활성화는 인증 통제가 아니다. 서버 측 검증을 기준으로 삼는다.

## 7. SR2-2 인증 수행 제한

### KISA 제안/근거

- 반복된 인증 시도를 제한하고 인증 실패 이력을 추적하도록 설계해야 한다. (p.93, p.346)
- 로그인 기능은 인증시도 횟수를 제한하고, 초과된 인증시도에 인증제한 정책을 적용해야 한다. (p.93)
- 로그인 시도 횟수는 3-5번 이내로 제한하고, 초과 시 추가 입력값 요구 또는 계정 잠금을 수행하도록 설계하는 예를 제시한다. (p.93-94)
- 사용자 ID별, 세션 ID별 로그인 횟수 추적을 위해 실패 횟수, 잠금 여부, 마지막 성공/실패 로그인 시간, 로그아웃 시간 등을 저장할 수 있도록 설계한다. (p.94)
- 실패한 인증시도를 로깅하여 허용되지 않은 로그인 시도를 분석할 수 있게 한다. (p.94)

### 설계 시 확인할 규격

- 실패 횟수를 계정 기준, IP 기준, 세션 기준, 디바이스 기준 중 무엇으로 추적할지 정한다.
- 잠금, 지연, CAPTCHA, 추가 인증 등 제한 정책을 정의한다.
- 실패 이력 저장 항목과 보존 기간을 정한다.
- 계정 존재 여부가 실패 메시지로 드러나지 않도록 `SR3-1`과 함께 메시지를 설계한다.
- 관리자 계정 또는 고위험 계정의 강화 정책을 별도로 둘지 검토한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 허용 실패 횟수.
- 잠금 시간 또는 해제 절차.
- 실패 이력 데이터 항목.
- 로깅 항목과 로그 마스킹 규칙.
- CAPTCHA 또는 추가 인증 적용 조건.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- KISA는 3-5회 이내 제한을 예시로 제시하지만, 최종 실패 횟수와 잠금 정책은 프로젝트 정책으로 확정한다.
- 계정 잠금은 서비스 거부 공격에 악용될 수 있으므로 위험도에 따라 지연, 추가 인증, 관리자 해제 등 대안을 함께 검토한다.
- 인증 실패 로그에 비밀번호, 토큰, 주민번호 등 민감값을 남기지 않는다.

## 8. SR2-3 비밀번호 관리

### KISA 제안/근거

- 비밀번호 생성규칙, 저장방법, 변경주기 등 비밀번호 관리 정책별 안전한 적용방법을 설계해야 한다. (p.96, p.346)
- 비밀번호 설정 시 KISA의 패스워드 보안 지침을 적용한다. (p.96, p.98)
- 네트워크로 비밀번호를 전송하는 경우 비밀번호를 암호화하거나 암호화된 통신 채널을 이용해야 한다. (p.96, p.99)
- 비밀번호 저장 시 솔트가 적용된 안전한 해시함수를 사용하고, 비밀번호 해시는 서버에서 실행해야 한다. (p.96, p.99)
- 비밀번호 재설정/변경 시 안전하게 변경할 수 있는 규칙을 정의해야 한다. (p.96, p.100)
- 비밀번호 관리 규칙을 정의해서 적용해야 한다. (p.96, p.100-101)

### 8.1 안전한 패스워드 생성 기준

KISA 원문은 안전한 패스워드 예로 다음 두 가지를 제시한다. (p.98)

- 알파벳 대문자, 알파벳 소문자, 특수문자, 숫자 등 4가지 문자 종류 중 두 종류 이상을 조합하고 8자리 이상으로 구성한다.
- 또는 10자리 이상으로 구성한다. 단, 숫자로만 구성하면 취약할 수 있다고 주의한다.

### 8.2 취약한 패스워드 예시

KISA 원문은 다음 유형을 안전하지 않은 패스워드로 든다. (p.98)

- 동일 문자 반복이나 `123123` 같은 특정 반복 패턴.
- `qwerty`처럼 키보드에서 연속한 위치의 문자열.
- 숫자가 맨 앞이나 뒤에 붙는 단순 구성.
- 가족, 이름, 생일, 주소, 휴대전화 번호 등 제3자가 알 수 있는 개인정보 기반 구성.
- 이용자 ID를 이용한 구성.
- 한글/영어 사전 단어 기반 구성.
- 사이트명, 기업명, 유명 인물명 등 널리 알려진 단어 포함.
- `O`를 `0`, `I`를 `1`처럼 비슷한 문자로 단순 치환한 형태.
- 시스템 초기값, 예제 비밀번호, 한글 발음을 영문으로 바꾸거나 영문 발음을 한글로 바꾼 형태.

### 8.3 비밀번호 전송 기준

- 웹브라우저와 웹서버, 서버와 서버 사이에서 비밀번호 같은 중요정보를 송수신할 때 TLS, VPN 등 통신 암호기술을 적용할 수 있다. (p.99)
- TLS 적용 시 표준 부합성, 상호호환성, 검증된 제품 여부, 오픈소스 사용 여부, 안전한 암호 알고리즘 적용 여부를 확인해야 한다. (p.99)
- 원문은 TLS 최신 버전 예로 TLS 1.2 및 TLS 1.3만 지원하고 안전한 암호 알고리즘 조합을 적용하라고 안내한다. (p.99)

### 8.4 비밀번호 저장 기준

- 비밀번호는 복호화되지 않는 일방향 해시 함수로 암호화하여 저장해야 한다. (p.99)
- 단순 SHA-256 같은 빠른 해시만으로 충분하다고 보지 않고, 솔트값을 추가하여 해시함수를 실행해야 한다. (p.99)
- 원문은 모든 패스워드가 고유 솔트를 갖고 솔트 길이가 32바이트 이상이면 솔트가 노출되지 않는 한 다이제스트 추측이 어렵다고 설명한다. (p.99)
- 비밀번호 해시는 클라이언트가 아니라 서버에서 실행해야 한다. (p.96)

### 8.5 비밀번호 변경 기준

- 사용자는 자신의 비밀번호가 제3자에게 노출되었을 경우 즉시 새 비밀번호로 변경해야 한다. (p.100)
- 변경 시 이전에 사용하지 않은 새 비밀번호를 사용해야 하며, 이전 비밀번호와 연관성이 없어야 한다. (p.100)
- 시스템은 사용자가 언제든지 비밀번호를 변경할 수 있는 기능을 제공하도록 설계한다. (p.100)
- 비밀번호 변경주기는 3개월 또는 6개월을 고려할 수 있다고 안내한다. (p.100-101)

### 8.6 비밀번호 재설정 기준

- 비밀번호 찾기 기능을 제공할 경우 이메일과 질의-답변으로 반드시 해당 사용자에게만 비밀번호 정보를 전달해야 한다. (p.100)
- 올바른 비밀번호가 입력되기 전에는 이메일 등 개인정보를 수정할 수 없도록 해야 한다. (p.100)
- 질의-답변 검증 시 일정 횟수 이상 정답을 맞히지 못하면 비밀번호 찾기 기능을 사용하지 못하도록 한다. (p.100)
- 검증 후 기존 비밀번호가 아니라 임시 비밀번호를 발급하도록 설계하고, 사용자는 즉시 새 비밀번호로 재설정해야 한다. (p.100)

### 8.7 비밀번호 관리 주기

- 개인 비밀번호는 사용자가 직접 생성하고, 그룹 비밀번호는 그룹의 장이 생성하여 구성원에게 안전한 방법으로 전달한다. (p.101)
- 비밀번호와 관련된 정보나 힌트를 제3자에게 제공하지 않아야 한다. (p.101)
- 시스템 및 소프트웨어의 초기 비밀번호는 설치 시 즉시 변경해야 한다. (p.101)
- 사용용도가 끝나거나 사용주기가 지난 비밀번호는 폐기한다. 사용자 계정 삭제와 함께 인증 비밀번호도 폐기한다. (p.101)

### 설계 시 확인할 규격

- 회원가입, 비밀번호 변경, 임시 비밀번호 발급, 관리자 초기 계정 생성 흐름을 모두 식별한다.
- 최소 길이, 문자 종류, 금지 패턴, 사용자 ID/개인정보 포함 금지, 이전 비밀번호 재사용 금지를 정의한다.
- 비밀번호 저장은 평문, 복호화 가능 암호화, 빠른 단순 해시가 아닌지 확인한다.
- 서버 측 해시, 고유 솔트, 안전한 해시/키 유도 알고리즘 적용 여부를 확인한다.
- 비밀번호 전송 구간이 HTTPS/TLS 등 안전한 채널인지 확인한다.
- 재설정 기능이 계정 탈취 경로가 되지 않도록 본인 확인, 시도 제한, 임시 비밀번호 만료, 즉시 변경 요구를 설계한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 최소 길이와 문자 조합 규칙.
- 금지 패턴 목록과 사용자 정보 비교 범위.
- 비밀번호 해시 알고리즘, 파라미터, 솔트 길이, pepper 사용 여부.
- 이전 비밀번호 재사용 제한 개수.
- 임시 비밀번호 만료 시간과 전달 채널.
- 재설정 검증 실패 제한 횟수.
- 비밀번호 변경주기 적용 여부와 예외 대상.
- 초기 비밀번호 발급 및 강제 변경 규칙.
- 비밀번호 관련 오류 메시지와 감사 로그 항목.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- `8~72`의 72처럼 특정 해시 라이브러리의 입력 길이 제한은 KISA 근거가 아니다. 사용 라이브러리의 제약으로 별도 확정한다.
- bcrypt, Argon2, PBKDF2 같은 구체 알고리즘과 비용 파라미터는 프로젝트 기술스택, 성능, 운영환경을 기준으로 확정한다.
- KISA 원문의 3개월 또는 6개월 변경주기는 고려 항목이다. 최신 정책상 주기적 강제 변경을 완화할 수 있는지 여부는 프로젝트 보안정책과 고객/감리 요구로 결정한다.
- 질의-답변 방식은 원문에 언급되지만 현대 서비스에서는 약한 본인확인 수단이 될 수 있으므로 적용 여부를 별도 위험평가로 결정한다.

## 9. SR2-4 중요자원 접근통제

### KISA 제안/근거

- 프로그램 설정, 민감한 사용자 데이터 등 중요자원을 정의하고, 신뢰할 수 있는 접근통제 방법과 접근통제 실패 시 대응방안을 설계해야 한다. (p.103, p.346)
- 중요자원, 중요기능, 관리자 페이지에 대한 접근통제 정책을 수립하여 적용해야 한다. (p.103)
- RBAC, ACL 등을 사용하여 사용자 역할이나 자원별 접근권한을 설계할 수 있다. (p.105-106)
- 접근권한은 최소권한으로 설정하고, 사용자별 또는 그룹별 접근을 체크한다. (p.106)
- 중요기능은 익명, 일반, 특권사용자, 관리자 영역으로 구분하고 역할기반 접근통제 및 비즈니스 로직에 따라 접근통제되도록 설계한다. (p.106)
- 파라미터 변조로 인증이 우회되지 않도록 변조 확인 절차를 구현한다. (p.107)
- 관리자 페이지는 추측 어려운 URL, TLS, 허용 IP, 별도 포트, 추가인증을 고려한다. (p.107)

### 설계 시 확인할 규격

- 중요자원 목록: 개인정보, 인증정보, 관리자 기능, 설정, 파일, DB 객체, 외부 연계 토큰.
- 역할/권한 모델: RBAC, ACL, 소유자 검증, 조직/부서 단위 권한.
- 객체 단위 권한 검증 지점: 상세 조회, 수정, 삭제, 다운로드, API 호출.
- 권한 실패 시 응답 코드, 사용자 메시지, 감사 로그.
- 관리자 페이지와 일반 사용자 페이지의 접근 경로와 인증/추가인증 조건.

### 프로젝트 보안가이드에서 확정해야 할 값

- 역할 코드와 권한 매트릭스.
- 중요자원 식별 목록과 보호 등급.
- 접근통제 적용 계층: 라우터, 미들웨어, 서비스, DB Row Level Security 등.
- 객체 소유권 검증 규칙.
- 관리자 접속 허용 조건: IP, MFA, VPN, 포트, 별도 도메인.
- 권한 실패 로그 항목과 보존 기간.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 관리자 URL을 숨기는 것만으로 접근통제가 완료되지 않는다.
- 클라이언트가 전달한 `role`, `userId`, `isAdmin` 같은 값은 권한 근거로 사용하지 않는다.
- 상위권한 수행이 필요한 경우 권한상승 시점과 원상복귀 기준을 별도 구현 규칙으로 둔다.

## 10. SR2-6 암호연산

### KISA 제안/근거

- 안전한 암호 알고리즘, 충분한 암호키 길이, 솔트, 충분한 난수값을 적용한 암호연산 수행방법을 설계해야 한다. (p.115, p.347)
- 대칭키 또는 비대칭키 암호화는 KISA 암호 알고리즘 및 키 길이 이용 안내서에서 정의한 알고리즘과 안전한 키 길이를 사용해야 한다. (p.115, p.117)
- 해시함수를 사용하는 경우 안전한 해시 알고리즘과 솔트값을 적용해야 한다. (p.115, p.119)
- 난수 생성 시 안전한 난수 생성 알고리즘을 사용해야 한다. (p.115, p.119)
- 원문은 2030년 이후 보안강도 128비트 이상, RSA 예시 3072비트 이상, ECC 예시 256비트 이상 등을 표로 제시한다. (p.117-118)
- 난수는 FIPS 140-2 인증 암호모듈의 난수생성기와 256비트 이상의 시드를 사용하는 방식을 설명한다. (p.119)

### 설계 시 확인할 규격

- 암호화 대상 데이터와 목적: 저장 암호화, 전송 암호화, 서명, 토큰, 비밀번호 해시, 난수.
- 알고리즘, 모드, 키 길이, IV/nonce, salt, random source를 구분해 정의한다.
- 금지 알고리즘과 허용 알고리즘을 명시한다.
- 암호키 관리는 `SR2-5`와 연결하고, 저장/전송 보호는 `SR2-7`, `SR2-8`과 연결한다.
- 난수 사용 위치: 세션 ID, 토큰, 임시 비밀번호, nonce, CSRF token.

### 프로젝트 보안가이드에서 확정해야 할 값

- 허용 암호 알고리즘과 최소 키 길이.
- 비밀번호 해시 또는 키 유도 알고리즘과 파라미터.
- 난수 생성 API.
- salt, IV, nonce 길이와 생성 방식.
- 금지 알고리즘: 예를 들어 MD5, SHA-1 단독 사용, DES, ECB mode 등.
- 암호 라이브러리와 검증된 모듈 사용 여부.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 원문 표는 기준 시점이 2021년 문서에 포함된 권고다. 새 프로젝트 시작 시 최신 KISA/NIST 기준을 다시 확인한다.
- Base64, URL encoding, compression은 암호화가 아니다.
- 난수는 일반 `random()`류 API가 아니라 암호학적으로 안전한 API를 사용하도록 프로젝트 언어별로 확정한다.

## 11. SR2-7 중요정보 저장

### KISA 제안/근거

- 비밀번호, 개인정보 등 중요정보 저장 시 안전한 저장 및 보관 방법을 설계해야 한다. (p.121, p.347)
- 중요정보 또는 개인정보는 암호화해서 저장해야 한다. (p.121-122)
- 중요정보가 다뤄지는 안전영역을 설정하고, 해당 영역 밖으로 중요정보가 누출되지 않도록 설계한다. (p.122)
- 서버 DB나 파일에 저장되는 중요정보는 안전한 암호 알고리즘과 암호키를 사용해 암호화한다. (p.122)
- 쿠키, HTML5 로컬저장소 같은 클라이언트 측 하드 드라이브에는 중요정보를 저장하지 않도록 설계하고, 불가피하면 암호화한다. (p.122)
- 불필요하거나 사용하지 않는 중요정보가 메모리에 남지 않도록 해야 한다. (p.121, p.123)
- 민감정보 페이지는 제한적인 캐시정책을 수립하고, 인증정보 웹 폼은 자동완성을 비활성화하도록 시큐어코딩 규칙을 정의한다. (p.123)

### 설계 시 확인할 규격

- 저장 대상 데이터의 보안 분류: 인증정보, 개인정보, 민감정보, 시스템정보.
- 저장 위치: DB, 파일, 로그, 캐시, 세션 저장소, 브라우저 저장소, 쿠키.
- 평문 저장 금지 대상과 암호화/해시/마스킹 방식.
- 메모리, 캐시, 로그, 임시파일에 중요정보가 남는지 확인한다.
- 클라이언트 저장소 사용 여부와 금지 데이터 목록을 정의한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 중요정보 분류표와 필드별 보호 방식.
- DB 컬럼 암호화, 파일 암호화, secret storage 사용 기준.
- 로그 마스킹 규칙.
- 브라우저 저장소/쿠키 저장 금지 목록.
- 캐시 제어 헤더와 autocomplete 정책.
- 데이터 삭제/폐기 및 메모리 초기화가 필요한 처리 위치.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 모든 개인정보를 같은 방식으로 암호화할지, 검색/정렬이 필요한 필드는 별도 토큰화/부분마스킹을 적용할지 정해야 한다.
- 쿠키에 중요정보를 넣지 않는 것이 우선이다. 불가피한 경우 암호화뿐 아니라 `SR2-8`, `SR4-1`의 쿠키 전송/세션 속성을 함께 적용한다.
- 로그와 분석 이벤트도 저장소다. 중요정보 저장 검토 범위에 포함한다.

## 12. SR2-8 중요정보 전송

### KISA 제안/근거

- 비밀번호, 개인정보, 쿠키 등 중요정보 전송 시 안전한 전송방법을 설계해야 한다. (p.125, p.347)
- 인증정보와 같은 민감정보 전송 시 안전하게 암호화해서 전송해야 한다. (p.125)
- 중요정보를 네트워크로 전송해야 하는 경우 안전한 암호모듈로 암호화하거나 안전한 통신 채널을 사용한다. (p.125)
- hidden 필드, Ajax 변수, 쿠키, 헤더, HTTP 응답헤더 등에 불필요하게 많은 데이터가 포함되어 전송되지 않도록 설계한다. (p.126)
- 쿠키에는 중요정보가 포함되지 않도록 설계하고, 불가피하면 세션쿠키로 설정하며 중요정보는 암호화해서 전송해야 한다. (p.126)

### 설계 시 확인할 규격

- 중요정보가 이동하는 모든 구간: 브라우저-서버, 서버-서버, 배치, 웹훅, 내부 API.
- TLS 적용 대상과 예외 없는 HTTPS 강제 여부.
- hidden field, query string, header, cookie, response body에 중요정보가 노출되는지 확인한다.
- 쿠키 전송 속성과 세션쿠키 여부를 `SR4-1`과 함께 설계한다.
- 외부 연계 시스템으로 전달하는 개인정보/인증정보 최소화 원칙을 적용한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- HTTPS/TLS 강제 정책과 허용 TLS 버전.
- 쿠키 `Secure`, `HttpOnly`, `SameSite`, domain, path, max-age 정책.
- URL query string에 포함 금지할 데이터 목록.
- API 요청/응답에서 마스킹 또는 제외할 필드.
- 외부 연계 전송 암호화 방식과 인증 방식.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- KISA 원문은 안전한 통신 채널 사용을 요구하지만, 실제 허용 TLS 버전과 cipher suite는 최신 운영 기준으로 확정해야 한다.
- 브라우저 개발자도구에 보이는 hidden field나 클라이언트 변수는 보호된 저장소가 아니다.
- 쿠키 암호화만으로 세션 탈취 방지가 완료되지 않는다. `Secure`, `HttpOnly`, `SameSite`, 세션 만료와 재발급 정책을 함께 둔다.

## 13. SR3-1 예외처리

### KISA 제안/근거

- 오류메시지에 개인정보, 시스템 정보, 민감정보 등이 노출되거나 부적절한 에러 처리로 의도치 않은 상황이 발생하지 않도록 설계해야 한다. (p.128, p.348)
- 명시적인 예외는 예외처리 블록을 이용하여 예외 발생 시 수행해야 하는 기능이 구현되도록 한다. (p.128-129)
- 런타임 예외는 입력값 범위를 체크하여 정상 동작 가능한 값만 사용되도록 보장한다. (p.128-129)
- 에러 발생 시 상세한 에러 정보가 사용자에게 노출되지 않게 한다. (p.128, p.130)
- 지정된 에러페이지로 리다이렉트하거나 서버 설정으로 특정 에러/예외에 지정 페이지가 보이도록 설계한다. (p.130)
- 사용자에게 보내는 오류메시지에 개인정보, 시스템정보, 민감정보가 포함되지 않도록 시큐어코딩 규칙을 정의한다. (p.130)

### 설계 시 확인할 규격

- 사용자용 오류 메시지와 내부 로그 메시지를 분리한다.
- 400/401/403/404/409/500 등 오류 코드별 공통 응답 형식을 정의한다.
- 스택트레이스, SQL, 파일 경로, 환경변수, 시스템 버전, 계정 존재 여부가 사용자 응답에 노출되지 않도록 한다.
- 예외 발생 시 보상 처리, 트랜잭션 롤백, 감사 로그 필요 여부를 정한다.
- 인증/권한 실패 메시지는 `SR2-1`, `SR2-2`, `SR2-4`와 함께 검토한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 공통 오류 응답 스키마.
- 사용자 메시지 문구.
- 내부 로그 항목과 마스킹 규칙.
- 예외 유형별 처리 정책.
- 운영/개발 환경별 디버그 출력 제한.
- 에러 페이지 또는 API 오류 처리 컴포넌트.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- 로그에는 원인 분석에 필요한 정보가 있어야 하지만, 민감정보를 그대로 남기면 `SR2-7` 위반이 될 수 있다.
- 인증 실패 메시지는 사용자 편의와 계정 열거 방지 사이에서 프로젝트 정책을 확정한다.
- 개발 환경의 상세 오류 출력이 운영 환경에 배포되지 않도록 설정 검증을 테스트에 포함한다.

## 14. SR4-1 세션통제

### KISA 제안/근거

- HTTP 연결 유지 시 세션을 안전하게 할당하고 관리하여 세션정보 노출과 세션 하이재킹을 방지하도록 설계해야 한다. (p.38, p.133, p.348)
- 다른 세션 간 데이터 공유 금지, 세션 ID 노출 금지, 로그인/재로그인 시 세션 ID 변경, 세션 종료 처리를 설계해야 한다. (p.38, p.133)
- 세션 간 데이터가 공유되지 않도록 웹 컨트롤러나 싱글톤 서비스 컴포넌트에서 읽고 쓰기 가능한 멤버 변수를 사용하지 않도록 설계한다. (p.134)
- 모든 페이지에서 로그아웃이 가능하도록 UI를 설계하고, 로그아웃 시 세션을 완전히 제거하는 API를 사용하도록 한다. (p.134)
- 세션 타임아웃은 중요기능의 경우 2-5분, 위험도가 낮은 경우 15-30분으로 설정하는 예를 제시한다. (p.134)
- 일정 시간 사용되지 않는 세션 정보는 강제로 삭제되도록 설계한다. (p.135)
- 중복 로그인을 허용하지 않는 경우 새 로그인 시 이전 세션 종료 또는 새 세션 차단 정책을 고려한다. (p.135)
- 세션 ID가 포함된 쿠키는 `HttpOnly`를 설정하여 JavaScript 조회를 막고 XSS 공격에 대응하도록 설계한다. (p.135)
- 사용자가 비밀번호를 변경하면 현재 활성 세션을 삭제하고 다시 할당한다. (p.135)
- 세션 ID는 안전한 서버에서 생성하고, 최소 128비트 길이와 안전한 난수 알고리즘을 적용해 예측 불가능해야 한다. (p.135)
- URL Rewrite로 세션 ID가 URL에 노출될 수 있으므로 사용하지 않도록 설계한다. (p.135)
- 로그인 성공 시 로그인 전 세션 ID는 파기하고 새 값으로 재할당하여 세션 고정 공격에 대응한다. (p.135)
- 장기간 접속 세션은 일정 주기로 세션 ID를 재할당하도록 설계한다. (p.135)

### 설계 시 확인할 규격

- 세션 저장소와 세션 ID 생성 주체가 서버인지 확인한다.
- 로그인 성공, 재로그인, 권한 상승, 비밀번호 변경, 로그아웃, 타임아웃에서 세션 처리 흐름을 정의한다.
- 쿠키 속성은 `SR2-8` 중요정보 전송과 함께 정한다.
- 싱글톤/전역 상태/캐시에 사용자별 데이터가 섞이지 않는지 점검한다.
- 중복 로그인 허용 여부와 기존 세션 처리 방식을 정한다.

### 프로젝트 보안가이드에서 확정해야 할 값

- 세션 타임아웃: 중요기능, 일반기능, 관리자별 시간.
- 세션 ID 길이와 난수 생성 방식.
- 세션 ID 재발급 시점.
- 로그아웃 처리 방식과 서버 세션 삭제 API.
- 중복 로그인 정책.
- 쿠키 `HttpOnly`, `Secure`, `SameSite`, domain, path, max-age 값.
- 비밀번호 변경/재설정 후 기존 세션 무효화 범위.

### 주의: 프로젝트 정책으로 별도 결정해야 하는 값

- KISA의 2-5분, 15-30분은 위험도별 예시다. 실제 타임아웃은 업무 특성, 사용자 경험, 고객 보안정책에 맞춰 확정한다.
- SPA/API 기반 앱도 세션 또는 토큰 수명, 재발급, 무효화 정책을 세션통제 범위로 다룬다.
- JWT처럼 서버 저장소가 없는 토큰을 사용할 경우 로그아웃과 강제 무효화를 어떻게 구현할지 별도 설계가 필요하다.

## 15. OWASP/CWE 보조 매핑 기준

KISA `SR-ID`를 보안가이드의 1차 준수 근거로 사용한다.
OWASP Top 10과 CWE는 위험군 또는 코드 약점 분류를 설명하는 보조값으로만 사용한다.

| KISA 항목 | OWASP 후보 | CWE 후보 | 매핑 주의 |
| --- | --- | --- | --- |
| `SR1-1` 입력값 검증 및 DB 조회 | OWASP-T10-2021 A03 Injection | CWE-89, CWE-20 | SQL Injection은 CWE-89가 직접 대응한다. 일반 입력 검증은 CWE-20을 보조로 사용한다. |
| `SR1-5` 출력값 검증 및 XSS | OWASP-T10-2021 A03 Injection | CWE-79, CWE-116 | XSS는 A03에 포함된다. 출력 인코딩 규칙은 CWE-116을 보조로 사용한다. |
| `SR2-1` 인증 대상 및 방식 | OWASP-T10-2021 A07 Identification and Authentication Failures | CWE-287, CWE-306 | 인증 실패는 CWE-287, 인증 누락은 CWE-306으로 세분화한다. |
| `SR2-2` 인증 수행 제한 | OWASP-T10-2021 A07 Identification and Authentication Failures | CWE-307 | 반복 인증 시도 제한과 직접 대응한다. |
| `SR2-3` 비밀번호 관리 | OWASP-T10-2021 A07, 저장 보호는 A02 Cryptographic Failures | CWE-521, CWE-916, CWE-759 | 정책, 저장, 변경/재설정 규격을 가능하면 별도 `SEC-ID`로 나눈다. |
| `SR2-4` 중요자원 접근통제 | OWASP-T10-2021 A01 Broken Access Control | CWE-862, CWE-863, CWE-639 | 기능 권한과 객체 소유권 검증을 구분한다. |
| `SR2-6` 암호연산 | OWASP-T10-2021 A02 Cryptographic Failures | CWE-327, CWE-326, CWE-330, CWE-338, CWE-916 | 암호화, 해시, 난수를 하나로 뭉치지 말고 구현 규격을 분리한다. |
| `SR2-7` 중요정보 저장 | OWASP-T10-2021 A02 Cryptographic Failures | CWE-312, CWE-922 | 비밀번호 저장이면 CWE-916도 함께 검토한다. |
| `SR2-8` 중요정보 전송 | OWASP-T10-2021 A02 Cryptographic Failures | CWE-319, CWE-523, CWE-614 | 쿠키 전송 속성은 `SR4-1`과 함께 검토한다. |
| `SR3-1` 예외처리 | OWASP-T10-2021 A05 Security Misconfiguration | CWE-209, CWE-532 | 사용자 오류 메시지 노출은 CWE-209, 로그 민감정보는 CWE-532로 분리한다. |
| `SR4-1` 세션통제 | OWASP-T10-2021 A07, 필요 시 A01/A02 | CWE-384, CWE-613, CWE-614, CWE-1275 | 세션 고정, 만료, 쿠키 속성, 토큰 저장 방식을 구분한다. |

보안가이드에는 다음처럼 1차 근거와 보조 분류를 분리해 적는다.

```yaml
reference_basis:
  primary:
    - KISA-SD-2021 SR2-4
classification:
  owasp:
    - OWASP-T10-2021 A01
  cwe:
    - CWE-862
    - CWE-639
mapping_note: "KISA SR2-4가 1차 준수 기준이며, OWASP/CWE는 접근통제 실패 유형을 보조 분류한다."
confidence: high
```

## 16. 보안가이드 작성 체크리스트

프로젝트 보안가이드의 각 `SEC-ID`는 최소한 다음 값을 가져야 한다.

| 구분 | 확인 항목 |
| --- | --- |
| 참조 | `KISA-SD-2021 SR-ID`, 필요 시 OWASP/CWE |
| 적용 대상 | `SCR`, `PGM`, `DB`, `IF` 중 하나 이상 |
| 보호 대상 | 인증정보, 개인정보, 민감정보, 시스템정보, 중요기능, 중요자원 |
| 정책값 | 길이, 횟수, 시간, 허용 목록, 금지 목록, 알고리즘, 오류 메시지 |
| 구현 위치 | 미들웨어, 서비스, 저장소, DB, 프론트엔드 컴포넌트, 배포 설정 |
| 실패 처리 | 사용자 응답, 로그, 감사 이벤트, 계정 잠금, 세션 무효화 |
| 검증 | `UT`, `IT`, `UI`, 보안리뷰 증적 중 하나 이상 |
| 미확정값 | KISA 근거가 아닌 프로젝트 결정값은 `확정필요`로 표시 |

## 17. KISA 근거로 쓰면 안 되는 값

다음은 KISA 원문에서 직접 확정한 값으로 쓰지 않는다. 필요하면 프로젝트 정책으로 별도 확정한다.

- 특정 라이브러리의 비밀번호 입력 최대 길이. 예: bcrypt 계열의 `72` 바이트 제한.
