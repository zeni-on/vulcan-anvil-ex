# Vulcan-Anvil Ex 데이터 표준 규칙

> 상태: 초안 v0.1
> 목적: 프로젝트 단어사전, 용어사전, 도메인, 데이터 항목명을 일관되게 관리하기 위한 규칙을 정의한다.

## 1. 기본 원칙

Vulcan-Anvil Ex는 데이터 설계와 프로그램 설계에서 사용하는 용어를 프로젝트 단어사전으로 관리한다.

기본 원칙:

- 공공데이터 공통표준에 존재하는 용어, 단어, 도메인은 우선 검토한다.
- 표준에 없는 용어는 프로젝트 단어사전에 등록하고 등록 사유를 남긴다.
- 화면 항목명, API 필드명, DB 컬럼명, 문서 용어가 서로 다를 수 있으므로 매핑을 명시한다.
- 개인정보, 인증정보, 식별정보, 민감정보는 보안항목 `SEC`와 연결한다.
- 데이터 표준 검토 결과는 DB명세서, 프로그램명세서, 화면설계서, 요구사항추적표에 반영한다.

## 2. 참조 표준

기본 참조 표준:

```text
PUBLIC-DATA-STD-2025-11
공공데이터 공통표준(2025.11월)
```

참조 시트:

| 시트 | 용도 |
| --- | --- |
| 공통표준용어 | 업무 용어, 데이터 항목명, 영문 약어명, 도메인 검토 |
| 공통표준단어 | 용어를 구성하는 단어와 영문 약어 검토 |
| 공통표준도메인 | 데이터 타입, 길이, 소수점, 저장/표현 형식 검토 |

## 3. 프로젝트 단어사전 구성

프로젝트 단어사전은 최소한 다음 정보를 가진다.

| 항목 | 설명 |
| --- | --- |
| 용어 ID | 프로젝트 내부 용어 ID |
| 한글명 | 사람이 읽는 용어명 |
| 영문명 | 영문 전체명 |
| 영문 약어 | DB 컬럼명/API 필드명 등에 사용할 약어 |
| 정의 | 프로젝트 안에서 사용하는 의미 |
| 도메인 | 데이터 타입, 길이, 형식 |
| 출처 | 공공데이터 공통표준 또는 프로젝트 |
| 표준 준용 여부 | 준용, 변형, 프로젝트신규, 확인필요 |
| 등록 사유 | 표준에 없거나 변형한 이유 |
| 관련 ID | `REQ`, `NREQ`, `DB`, `PGM`, `SCR`, `SEC` 등 |

## 4. 용어 ID

프로젝트 단어사전의 용어 ID는 다음 형식을 사용한다.

```text
TERM-001
WORD-001
DOMAIN-001
```

구분:

| Prefix | 의미 |
| --- | --- |
| `TERM` | 업무 용어 또는 데이터 항목 용어 |
| `WORD` | 용어를 구성하는 단어 |
| `DOMAIN` | 데이터 도메인 |

`TERM`, `WORD`, `DOMAIN`은 프로젝트 단어사전 안에서만 사용하는 보조 ID이며, 요구사항 추적의 중심 ID는 아니다.

## 5. 표준 준용 상태

| 상태 | 의미 |
| --- | --- |
| `준용` | 공공데이터 공통표준의 용어/단어/도메인을 그대로 사용 |
| `변형` | 표준을 참고하되 프로젝트 의미에 맞게 일부 변경 |
| `프로젝트신규` | 표준에 없어서 프로젝트에서 새로 정의 |
| `확인필요` | 표준 존재 여부 또는 적용 여부 확인 필요 |
| `제외` | 표준 검토 대상이 아니거나 적용하지 않음 |

## 6. 명명 규칙

기본 명명 규칙:

- 한글명은 업무 사용자가 이해할 수 있는 이름으로 작성한다.
- 영문명은 의미가 드러나는 전체 이름을 사용한다.
- 영문 약어는 공공데이터 공통표준 약어를 우선 사용한다.
- DB 컬럼명은 프로젝트의 DB 명명 규칙에 맞춰 `snake_case` 또는 `UPPER_SNAKE_CASE` 중 하나로 통일한다.
- API 필드명은 프로젝트의 API 규칙에 맞춰 `camelCase` 또는 `snake_case` 중 하나로 통일한다.
- Boolean 값은 의미가 명확한 이름을 사용하고 `YN`, `is`, `has` 등 프로젝트 규칙을 문서화한다.

## 7. 화면/API/DB 매핑

하나의 데이터 항목은 화면, API, DB에서 다른 이름으로 표현될 수 있다.

예시:

| TERM-ID | 한글명 | 화면 항목명 | API 필드명 | DB 컬럼명 | 도메인 |
| --- | --- | --- | --- | --- | --- |
| TERM-001 | 사용자명 | 이름 | userName | user_nm | 명V100 |

규칙:

- 화면 항목명은 사용자 이해를 우선한다.
- API 필드명은 개발자 사용성과 일관성을 우선한다.
- DB 컬럼명은 데이터 표준과 물리 모델 규칙을 우선한다.
- 서로 다른 이름을 쓰는 경우 매핑 사유를 남긴다.

## 8. 보안 분류

데이터 항목은 보안/개인정보 관점의 분류를 가져야 한다.

| 분류 | 의미 |
| --- | --- |
| 일반 | 특별한 보호가 필요하지 않은 일반 데이터 |
| 식별정보 | 사용자를 식별할 수 있는 정보 |
| 인증정보 | 비밀번호, 토큰, 인증서 등 인증에 사용하는 정보 |
| 개인정보 | 개인정보 보호 대상 정보 |
| 민감정보 | 특별히 보호가 필요한 민감 정보 |
| 시스템정보 | 시스템 내부 구조, 설정, 경로, 키 등 |

규칙:

- `인증정보`, `개인정보`, `민감정보`, `시스템정보`는 관련 `SEC` 항목과 연결한다.
- 저장/전송/마스킹/로그 출력 정책을 프로그램명세서 또는 DB명세서에 명시한다.

## 9. 에이전트 작성 기준

에이전트가 데이터 항목을 작성할 때는 다음 순서를 따른다.

1. 요구사항, 기능명세, 화면설계, 프로그램명세에서 데이터 항목 후보를 수집한다.
2. 공공데이터 공통표준에서 동일하거나 유사한 용어를 찾는다.
3. 표준 용어가 있으면 영문 약어와 도메인을 검토한다.
4. 표준 용어가 없으면 프로젝트 신규 용어로 등록하고 등록 사유를 작성한다.
5. 화면/API/DB 이름 매핑을 작성한다.
6. 개인정보/인증정보/민감정보 여부를 분류한다.
7. 필요한 경우 `SEC` 항목과 연결한다.

## 10. 검토 기준

| 항목 | 확인 |
| --- | --- |
| 주요 데이터 항목이 프로젝트 단어사전에 등록되었는가 |  |
| 공공데이터 공통표준 검토 여부가 기록되었는가 |  |
| 표준에 없는 용어의 등록 사유가 기록되었는가 |  |
| 화면/API/DB 항목명 매핑이 존재하는가 |  |
| 개인정보/인증정보/민감정보가 분류되었는가 |  |
| 보호가 필요한 데이터 항목이 `SEC`와 연결되었는가 |  |
