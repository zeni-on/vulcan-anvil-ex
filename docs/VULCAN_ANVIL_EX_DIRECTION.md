# Vulcan-Anvil Ex 방향성 초안

> 작성일: 2026-04-27
> 목적: Vulcan-Anvil Ex를 특정 모델에 종속되지 않는 AI 협업 기반 개발 기준 프레임워크로 재정의한다. 개인 프로젝트부터 SI/감리 대응 프로젝트까지 같은 Core를 공유하되, 프로젝트 성격에 따라 프로세스 강도를 조절한다.

## 1. 기본 방향

Vulcan-Anvil Ex는 기존 Vulcan-Anvil에서 출발한다. 기존 Vulcan-Anvil은 Claude Code에 맞춰 설계되었지만, Ex는 Claude 전용 프레임워크나 Codex 전용 프레임워크가 되어서는 안 된다.

Ex의 목표는 여러 AI 도구와 사람이 함께 일하더라도 프로젝트 산출물, 추적성, 승인 흐름, 검증 기준을 하나의 규격으로 유지하는 것이다. 이 규격은 대형 SI에만 필요한 것이 아니라, 개인 앱이나 작은 제품을 만들 때도 프로젝트 기억을 유지하고 변경을 안전하게 쌓기 위한 개발 기준으로 사용할 수 있어야 한다.

Ex의 기본 실행 단위는 사람의 수작업이 아니라 에이전트와의 협업이다. 요구사항 정리, 설계 초안 작성, 테스트 계획 수립, 구현, 리뷰, 문서 갱신, 추적성 점검은 에이전트가 수행할 수 있어야 한다. 다만 좋은 결과는 단순한 자동 실행에서 나오지 않는다. 특히 Phase 0와 상위 설계 단계에서는 사용자가 메인 에이전트와 오래 대화하며 아이디어를 꺼내고, 문제를 다시 정의하고, 기술 선택과 구현 방향을 함께 다듬는 시간이 중요하다.

사람은 모든 산출물을 직접 작성하는 주체라기보다, 문제의 맥락과 의도를 제공하고, 에이전트와 함께 방향을 탐색하며, 중요한 판단을 승인하고 최종 검수를 수행하는 역할을 맡는다.

핵심 문장:

> 모델은 자유롭게, 산출물은 추적 가능하게.

팀원은 Claude, Codex, Cursor, Copilot 등 각자 익숙한 도구를 사용할 수 있다. 다만 결과물은 Vulcan-Anvil Ex의 공통 프로세스와 문서 규격을 따라야 한다.

따라서 Ex는 차가운 자동화 파이프라인이 아니라, 사용자와 메인 에이전트가 팀을 이루어 프로젝트를 키워가는 방식에 가깝다. 에이전트는 단순 작업자가 아니라 아이디어 파트너, 설계 파트너, 구현 파트너가 될 수 있어야 한다. Core는 그 협업이 대화창 안에서 흩어지지 않고 산출물과 추적성으로 남도록 돕는다.

## 2. Ex가 필요한 이유

Codex나 Claude 같은 AI 코딩 에이전트는 이미 많은 개발 작업을 잘 수행한다. 코드베이스 탐색, 계획 수립, 구현, 테스트 실행, 리뷰 보조 같은 일은 각 도구가 어느 정도 자체적으로 지원한다.

하지만 좋은 개발에는 단순한 코드 작성 능력만으로 부족한 부분이 있다. 개인 프로젝트든 팀 프로젝트든, 시간이 지나면 왜 그렇게 만들었는지, 어떤 요구를 기준으로 구현했는지, 어떤 검증을 했는지 잊기 쉽다. 프로젝트 차원의 기억과 기준이 필요하다.

Ex가 해결하려는 문제는 다음과 같다.

- 요구사항과 인수 기준이 대화창 밖에서도 유지되어야 한다.
- 요구사항, 설계, 테스트, 리뷰, 코드, 증적이 서로 연결되어야 한다.
- Gate 전환과 사람의 승인 지점이 명확해야 한다.
- 변경 요청이 들어왔을 때 Backlog와 Rollback 범위가 관리되어야 한다.
- 프로젝트 성격에 따라 문서 메타데이터, 버전, 승인 이력, 감리 대응 기록이 남아야 한다.
- PM, 개발자, QA, 리뷰어, 감리자가 같은 상태를 볼 수 있어야 한다.

따라서 Ex는 AI 모델 자체를 대체하는 것이 아니라, 여러 AI 도구가 작업하더라도 프로젝트가 흐트러지지 않도록 잡아주는 운영 체계가 되어야 한다.

Ex가 전제하는 개발 철학:

> 문서 없이 개발하지 않는다. 다만 모든 문서가 항상 무거울 필요는 없다.

## 3. Core와 Adapter 분리

다음 버전의 핵심 구조는 Core와 Adapter의 분리다.

### Vulcan-Anvil Ex Core

Core는 특정 AI 모델이나 도구와 무관한 공통 표준이다.

Core가 담당할 내용:

- Phase와 Gate 모델
- 산출물 문서 규격
- REQ, AC, TST, UT, SEC, BL 식별자 규칙
- 요구사항 추적 매트릭스 규칙
- Backlog 분류와 증분 Rollback 규칙
- 감리 대응 메타데이터와 승인 규칙
- `check-trace` 검증 기준
- Dashboard와 보고용 snapshot/export 스키마

Core는 Ex의 중심이다. Claude나 Codex는 Core를 실행하는 여러 방법 중 하나일 뿐이다.

### Adapter

Adapter는 Core를 각 AI 도구 환경에 맞게 적용하는 층이다.

초기 Adapter 후보:

- `adapter-claude`: 기존 `.claude/agents`, `.claude/rules`, `.claude/skills`
- `adapter-codex`: Codex용 가이드, skill/plugin 후보, 작업 playbook

향후에는 Cursor, Copilot, Gemini, 사내 에이전트용 Adapter도 추가할 수 있다.

중요한 점은 `.claude`가 Ex의 본체가 아니라는 것이다. `.claude`는 Claude용 Adapter이고, Ex의 본체는 Core 규격이어야 한다.

## 4. 역할 용어 원칙

기존 Vulcan-Anvil은 `PM`, `QA`, `Architect`처럼 실제 조직 직책에 가까운 이름을 Persona 이름으로 사용했다. 이 방식은 이해하기 쉽지만, Ex의 Core 표준으로 그대로 가져가면 오해가 생길 수 있다.

예를 들어 요구사항 정의 단계에서 `PM`이 산출물을 작성한다고 되어 있으면, 실제 프로젝트의 PM이 요구사항 정의만 담당하는 것처럼 보일 수 있다. 하지만 현실의 PM은 범위, 일정, 예산, 리스크, 커뮤니케이션, 승인 흐름까지 폭넓게 책임진다.

따라서 Ex의 Core에서는 직책명이 아니라 산출물 책임을 기준으로 역할명을 정의한다.

핵심 원칙:

> 역할명은 조직 직책이 아니라, 해당 산출물과 판단에 대한 책임을 나타낸다.

한 사람이 여러 역할을 수행할 수 있고, 하나의 AI 에이전트가 여러 역할을 수행할 수도 있다. 반대로 중요한 SI 프로젝트에서는 같은 역할을 사람과 AI가 함께 수행하고, 최종 판단은 사람이 승인할 수 있다.

Ex에서 역할은 반드시 사람에게만 배정되지 않는다. 기본적으로 각 역할은 에이전트가 수행 가능한 작업 단위로 정의되어야 한다. 사용자는 작업의 목적, 제약, 승인 기준을 제공하고, 에이전트는 해당 역할에 맞는 산출물을 작성하거나 갱신한다. 최종 Gate 승인과 사용자 관점의 검수는 사람이 수행하는 것을 원칙으로 한다.

권장 역할명:

| 기존 용어 | Core 권장 용어 | 의미 |
| --- | --- | --- |
| PM | Requirements Lead | Gate 1 요구사항, 인수 기준, 범위 정의 책임 |
| BA | Business Analyst | 업무 흐름, 사용자 관점, 비즈니스 요구 분석 |
| SA | Solution Architect | Phase 0 수준의 솔루션 방향과 기술 전략 |
| Architect | Technical Architect | Gate 2 기술 설계와 구조 판단 |
| DBA | Data Architect | 데이터 모델, 스키마, 마이그레이션 관점 설계 |
| UI Designer | UX/UI Designer | 화면 흐름, 정보 구조, 사용자 경험 설계 |
| QA (Gate 3) | Test Designer | 테스트 전략, 테스트 케이스, 검증 기준 설계 |
| QA (Gate 4) | QA Reviewer | 구현 결과 검증, 테스트 결과 리뷰, 품질 판정 |
| UX Reviewer | UX Reviewer 또는 Design QA | 구현된 UI/UX가 설계 의도와 맞는지 검수 |
| Frontend-dev | Frontend Engineer | UI 구현, 클라이언트 상태, 접근성, 화면 품질 |
| Backend-dev | Backend Engineer | API, 도메인 로직, 데이터 연동, 서버 품질 |
| Human | Approver | Gate 통과와 최종 승인 책임 |
| Estimator | Estimation Lead | 규모, 일정, 공수, 리스크 추정 |
| - | Documentation Curator | 문서 일관성, 용어, 버전, 산출물 품질 관리 |

기존 `.claude/agents`의 역할명은 Claude Adapter 안에서는 호환성을 위해 유지할 수 있다. 다만 Core 문서, Dashboard, 감리 대응 문서에서는 위와 같은 책임 기반 역할명을 사용하는 것이 바람직하다.

이렇게 하면 Ex는 특정 조직 구조나 특정 AI 도구에 묶이지 않는다. 개인 프로젝트에서는 한 사람이 `Requirements Lead`, `Technical Architect`, `Backend Engineer`, `QA Reviewer`, `Approver`를 모두 맡을 수 있고, SI 프로젝트에서는 각 역할을 실제 팀 구성에 맞게 분리할 수 있다.

에이전트 중심으로 보면 개인 프로젝트에서는 한 명의 사용자가 여러 에이전트 역할을 호출해 프로젝트를 진행할 수 있고, 팀 프로젝트에서는 각 팀원이 자신이 선호하는 AI 도구를 사용해 동일한 Core 산출물 규격을 채울 수 있다. 즉 역할은 사람의 직책이 아니라 에이전트가 수행해야 할 작업 모드이기도 하다.

## 5. 프로세스 강도 모드

Ex는 모든 작업에 무조건 무거운 Gate 프로세스를 강제하면 안 된다. 프로젝트와 변경의 성격에 따라 강도를 조절할 수 있어야 한다.

### Lightweight 모드

작은 수정이나 위험도가 낮은 변경에 사용한다.

- 문서 수정은 최소화하거나 생략 가능
- 구현과 집중 검증 중심
- 짧은 리뷰 메모만 남김
- 추적성 변경이 없으면 formal rollback 생략 가능

### Standard 모드

일반적인 기능 개발에 사용한다.

- 필요 시 요구사항 갱신
- 설계 문서 갱신
- 테스트 계획 갱신
- 구현
- QA 리뷰
- TRACEABILITY 갱신

### Strict 모드

SI, 감리, 보안, 규제, 대규모 변경에 사용한다.

- Gate 절차 엄격 적용
- 승인 메타데이터 필수
- 요구사항 추적성 필수
- 테스트 증빙 필수
- 리뷰 문서 필수
- Dashboard와 감리 제출 패키지 준비 상태 확인

이 구조를 두면 팀은 필요한 곳에만 절차를 무겁게 적용할 수 있다.

## 6. Dashboard의 역할

Ex에서 Dashboard는 부가 기능이 아니라 핵심 구성 요소가 되어야 한다.

현재 Dashboard는 이미 다음 기능을 갖고 있다.

- 여러 프로젝트 등록
- 로컬 프로젝트 데이터 소스 지원
- `session.json` 기반 Gate 상태 표시
- 산출물 문서 트리 표시
- Markdown 문서 뷰어
- 최근 커밋 타임라인
- 통계 카드 표시

Ex의 Dashboard는 앞으로 다음 방향으로 발전할 수 있다.

- 산출물 완성도 표시
- 필수 메타데이터 누락 경고
- REQ에서 Design, TST, Review로 이동하는 추적성 탐색
- TRACEABILITY 행 단위 drill-down
- Backlog와 Rollback 상태 표시
- 감리 제출 패키지 준비율 표시
- 어떤 모델이 작업했는지는 선택 메타데이터로만 표시

Dashboard가 답해야 하는 질문:

> 지금 프로젝트는 어디까지 왔고, 무엇이 부족하며, 어떤 증거가 남아 있는가?

에이전트 중심 운영에서 Dashboard는 사람을 위한 최종 관제 화면이 된다. 각 에이전트가 만든 요구사항, 설계, 테스트, 구현, 리뷰 결과를 사용자가 한곳에서 확인하고, 누락된 산출물이나 승인 대기 상태를 판단할 수 있어야 한다.

## 7. gstack과의 관계

Vulcan-Anvil Ex는 gstack류의 AI 개발 워크플로에서 배울 점이 있다.

gstack의 핵심 흐름은 다음과 같이 이해할 수 있다.

> Think -> Plan -> Build -> Review -> Test -> Ship -> Reflect

이 흐름은 AI와 함께 개발할 때 자연스러운 작업 리듬을 제공한다. 문제를 다시 생각하고, 계획을 세우고, 구현하고, 리뷰하고, 테스트하고, 배포하고, 회고하는 순서다.

Vulcan-Anvil Ex는 이 흐름을 그대로 복제하기보다, Ex의 Gate 체계와 산출물 중심 방식에 맞게 해석한다.

| gstack 흐름 | Vulcan-Anvil Ex 해석 |
| --- | --- |
| Think | Phase 0 / Discovery / 문제 재정의 |
| Plan | Gate 1 요구사항 + Gate 2 설계 + Gate 3 테스트 계획 |
| Build | 구현 |
| Review | UX 리뷰 + 코드 리뷰 |
| Test | QA 실행 + 테스트 증빙 |
| Ship | Gate 5 승인 + 릴리즈 |
| Reflect | Backlog + 회고 + 다음 증분 Rollback |

이때 Phase 0 / Think 단계는 단순한 사전 조사 단계가 아니다. 사용자가 메인 에이전트와 함께 아이디어를 발전시키고, 제품의 방향을 잡고, 가능한 기술 경로를 비교하고, 프로젝트의 언어를 만들어가는 협업 단계다. 이 단계에서의 팀워크 품질이 이후 요구사항, 설계, 구현 품질을 크게 좌우한다.

gstack은 역할 기반 작업 리듬과 에이전트 실행 흐름에 강점이 있다. Ex는 여기에 더해 프로젝트 산출물과 추적성을 중심에 둔다.

Ex의 차별점:

- 역할 프롬프트보다 산출물의 품질과 연결성을 우선한다.
- 작업 결과가 문서, 테스트, 리뷰, 증빙으로 남아야 한다.
- Dashboard에서 현재 상태와 누락 항목을 확인할 수 있어야 한다.
- 개인 프로젝트부터 SI/감리 프로젝트까지 같은 Core를 사용할 수 있어야 한다.

즉 gstack은 좋은 개발 리듬의 참고 모델이고, Vulcan-Anvil Ex는 그 리듬을 문서화된 프로젝트 운영 체계로 고정하는 방향을 지향한다.

## 8. 산출물 규격과 감리 대응

Ex의 산출물 규격은 모든 프로젝트에서 사용할 수 있어야 한다. 개인 프로젝트에서는 가볍게 적용하고, 실제 SI 프로젝트나 감리 대상 프로젝트에서는 더 엄격하게 적용한다.

산출물 템플릿은 이론만으로 새로 만들기보다, 실제 감리에서 사용했던 문서를 분석해 하나씩 정제한다. 이미 감리 대응에 사용된 문서는 어떤 항목이 필요한지, 어떤 증빙이 요구되는지, 어떤 항목이 형식적으로만 남기 쉬운지 알려주는 좋은 기준 자료가 된다.

Ex의 목표는 감리 문서를 그대로 복제하는 것이 아니다. 기존 문서를 기반으로 하되, 에이전트가 작성하고 사용자가 검토하기 좋은 구조로 다시 설계한다. 사람이 빈 양식을 수동으로 채우는 방식이 아니라, 대화, 코드, Git 이력, 테스트 결과, Dashboard 상태를 바탕으로 에이전트가 초안을 만들고 사용자가 최종 판단하는 방식이어야 한다.

특히 감리를 받는 SI 프로젝트에서는 각 핵심 산출물이 독립적으로 검토 가능한 메타데이터를 가져야 한다.

공통 메타데이터 후보:

- 프로젝트명
- 산출물 ID
- 산출물 유형
- 관련 REQ ID
- 버전
- 상태
- 작성자 역할
- 검토자 또는 승인자
- 작성일
- 수정일
- 변경 사유
- 승인일

문서 상태 후보:

- 초안
- 검토 요청
- 승인
- 구현 중
- 구현 완료
- 테스트 완료
- 최종 승인
- 대체됨
- 삭제됨

정확한 문서 규격은 프로젝트 규모와 목적에 따라 단계적으로 다듬는다. 개인 프로젝트에서는 최소 메타데이터만 사용하고, SI/감리 프로젝트에서는 승인자, 변경 사유, 증빙, 제출 상태까지 확장한다.

산출물 템플릿 구성은 Ex의 중요한 마일스톤이다. 이는 단순한 문서 양식 작업이 아니라, 개발 프로젝트가 어떤 기준으로 생각하고, 만들고, 검증하고, 남겨야 하는지를 정의하는 작업이다.

## 9. 기존 Vulcan-Anvil에서 Ex로 가는 방향

기존 Claude 중심 구현은 버릴 대상이 아니다. Ex에서는 이를 첫 번째 Adapter로 취급한다.

권장 전환 순서:

1. 기존 Vulcan-Anvil의 Git history를 유지한다.
2. 저장소와 README를 Vulcan-Anvil Ex 방향으로 리브랜딩한다.
3. Core 규격 문서를 추가한다.
4. 기존 Claude 자산을 Adapter 경계 아래로 이동하거나 복제한다.
5. Codex용 Adapter 가이드를 추가한다.
6. 감리 대응 문서 표준을 강화한다.
7. `vulcan.py`와 Dashboard를 Core 개념 중심으로 확장한다.
8. 기존 Vulcan-Anvil 프로젝트와 가능한 범위에서 하위 호환성을 유지한다.

## 10. 첫 번째 마일스톤

처음부터 모든 것을 바꾸기보다, 작고 명확한 첫 마일스톤을 잡는다.

문서 산출물:

- `CORE_SPEC.md`
- `ARTIFACT_STANDARD.md`
- `TRACEABILITY_STANDARD.md`
- `ADAPTER_GUIDE.md`
- `DASHBOARD_DIRECTION.md`
- README 리브랜딩 초안

엔지니어링 작업:

- 기존 Dashboard가 계속 동작하도록 유지
- 로컬 Vulcan 프로젝트 등록과 문서 열람 검증
- Dashboard가 기대하는 snapshot 스키마 정리
- 감리 대응에 필요한 최소 `check-trace` 규칙 식별

## 11. 운영 원칙

Vulcan-Anvil Ex는 팀원에게 특정 AI 도구 사용을 강제하지 않는다.

Ex가 강제해야 하는 것은 도구가 아니라 산출물의 품질과 추적성이다.

Ex는 에이전트가 실무 작업 대부분을 수행하는 것을 전제로 한다. 다만 프로젝트의 출발점과 중요한 전환점에서는 사용자와 메인 에이전트의 깊은 협업이 필요하다. 요구사항 정리부터 개발, 테스트, 문서 갱신까지 에이전트가 진행하되, 문제 정의와 방향 설정은 사용자와 에이전트가 함께 만들고, 최종 책임과 승인 권한은 사용자 또는 프로젝트의 지정된 승인자가 가진다.

운영 원칙:

> 어떤 AI 도구를 써도 된다. 하지만 결과물은 공유 가능하고, 검토 가능하고, 감리 가능한 형태로 남겨야 한다.

개인 프로젝트에서는 이 원칙을 더 가볍게 표현할 수 있다.

> 혼자 만들더라도, 나중의 내가 이해할 수 있게 남긴다.
