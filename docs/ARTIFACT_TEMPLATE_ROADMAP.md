# Vulcan-Anvil Ex 산출물 템플릿 로드맵

> 작성일: 2026-04-28
> 목적: 실제 감리 경험을 기반으로 Vulcan-Anvil Ex의 개발 프로젝트 산출물 템플릿을 단계적으로 구성한다.
> 상태: 초기 산출물 템플릿 구상 문서. 현재 우선순위와 다음 작업은 `docs/ROADMAP.md`를 기준으로 한다.

## 0. 현재 위치

이 문서는 Vulcan-Anvil Ex 초기 단계에서 감리 산출물 템플릿을 어떻게 구성할지 정리한 참고 문서다.

`0.4.0` 기준으로 요구사항정의서, SW Architecture, Program Design, API Spec, DB Spec, Security Guide, Test Case, Test Result, Traceability, Backlog, Release Approval 등 핵심 템플릿은 이미 `docs/templates/`와 `docs/core/`에 상당 부분 반영되어 있다.

따라서 이 문서의 "다음 작업"은 최신 실행 우선순위가 아니라, 산출물 템플릿 설계의 배경 메모로 본다. 현재 작업 우선순위는 `docs/ROADMAP.md`를 따른다.

## 1. 기본 방향

Vulcan-Anvil Ex의 산출물 템플릿은 책상 위에서 새로 만든 양식이 아니라, 실제 프로젝트와 감리 대응에서 검증된 문서를 기반으로 만든다.

목표는 기존 감리 문서를 그대로 옮기는 것이 아니다. 감리에 필요한 형식과 증빙 요건은 유지하되, 에이전트가 작성하고 사람이 검토하기 좋은 구조로 재설계한다.

핵심 원칙:

> 에이전트가 초안을 만들고, 사용자가 판단하고, 프로젝트는 추적 가능한 기록을 남긴다.

## 2. 템플릿 설계 원칙

각 산출물 템플릿은 다음 기준을 가져야 한다.

- 문서의 목적이 명확해야 한다.
- 어느 Gate에서 작성 또는 갱신되는지 명확해야 한다.
- 어떤 역할 또는 에이전트 모드가 책임지는지 명확해야 한다.
- 어떤 입력 자료를 기반으로 작성하는지 명확해야 한다.
- 어떤 ID와 연결되는지 명확해야 한다.
- 어떤 기준으로 검토하고 승인하는지 명확해야 한다.
- 개인 프로젝트와 SI/감리 프로젝트에서 적용 강도를 조절할 수 있어야 한다.

템플릿은 사람이 빈칸을 채우는 양식이 아니라, 에이전트가 대화, 코드, Git 이력, 테스트 결과, Dashboard 상태를 참고해 작성할 수 있는 구조여야 한다.

## 3. 산출물 분류 초안

### Phase 0 / Discovery

- 아이디어 노트
- 문제 정의서
- 사용자/업무 맥락 정리
- 기술 검토 메모
- 범위 후보 및 제외 범위
- 초기 리스크 메모

### Gate 1 / Requirements

- 요구사항정의서
- 인수 기준 정의서
- 요구사항 변경 이력
- 용어 정의서

### Gate 2 / Design

- 아키텍처 설계서
- 화면 설계서
- API 정의서
- 인터페이스 정의서
- 데이터 모델 정의서
- 테이블 정의서
- 보안 설계 메모

### Gate 3 / Test Planning

- 테스트 계획서
- 테스트 케이스 정의서
- 단위 테스트 기준
- 통합 테스트 기준
- 보안 테스트 기준

### Gate 4 / Verification

- 테스트 결과서
- QA 리뷰 보고서
- UX/UI 검수 결과서
- 결함 및 조치 이력
- 요구사항 추적 매트릭스

### Gate 5 / Approval and Release

- 최종 승인 기록
- 릴리즈 노트
- 배포 체크리스트
- 감리 제출 패키지 목록

### 상시 관리 문서

- 회의록
- 이슈 관리 대장
- 변경 관리 대장
- 의사결정 기록
- Backlog
- Rollback 기록

## 4. 문서별 공통 메타데이터

모든 주요 산출물은 최소한 다음 메타데이터를 가진다.

| 항목 | 설명 |
| --- | --- |
| 문서 ID | 산출물의 고유 식별자 |
| 문서명 | 사람이 읽는 문서 이름 |
| 프로젝트명 | 대상 프로젝트 |
| Gate | 작성 또는 갱신되는 Gate |
| 관련 ID | REQ, AC, TST, UT, SEC, BL 등 연결 ID |
| 책임 역할 | Core 기준 역할명 |
| 작성 주체 | 사람 또는 에이전트 |
| 검토자 | 검토 책임자 또는 역할 |
| 승인자 | 최종 승인 책임자 |
| 상태 | 초안, 검토 요청, 승인, 대체됨 등 |
| 버전 | 문서 버전 |
| 작성일 | 최초 작성일 |
| 수정일 | 마지막 수정일 |
| 변경 사유 | 주요 변경 이유 |

## 5. 에이전트 작성 기준

각 템플릿에는 에이전트가 문서를 작성할 때 필요한 지시가 포함되어야 한다.

예시:

- 어떤 질문을 사용자에게 먼저 해야 하는가?
- 어떤 기존 문서를 읽어야 하는가?
- 어떤 코드나 설정 파일을 확인해야 하는가?
- 어떤 항목은 추정하면 안 되고 사용자 확인이 필요한가?
- 어떤 항목은 Dashboard나 `check-trace`가 자동 검증할 수 있는가?
- 어떤 근거를 문서에 남겨야 하는가?

이 기준이 있어야 Claude, Codex, Cursor 등 서로 다른 도구를 쓰더라도 산출물의 품질과 구조가 흔들리지 않는다.

## 6. 우선순위

첫 번째 우선순위는 Gate 흐름의 중심 문서다.

1. 요구사항정의서
2. 아키텍처 설계서
3. 화면 설계서
4. API 정의서
5. 데이터 모델 정의서
6. 테스트 계획서
7. 테스트 결과서
8. 요구사항 추적 매트릭스
9. 변경 관리 대장
10. 최종 승인 기록

두 번째 우선순위는 감리 대응과 운영 문서다.

1. 회의록
2. 이슈 관리 대장
3. 의사결정 기록
4. 릴리즈 노트
5. 감리 제출 패키지 목록

## 7. 다음 작업

다음 세션에서는 작년에 감리 때 사용했던 산출물 목록을 기준으로, 실제 템플릿 후보를 하나씩 분석한다.

진행 방식:

1. 기존 산출물 목록을 수집한다.
2. 각 산출물을 Gate 기준으로 분류한다.
3. 실제 필요한 항목과 형식적인 항목을 구분한다.
4. 에이전트가 작성 가능한 구조로 재설계한다.
5. Strict 모드 기준 템플릿을 먼저 만든다.
6. 이후 Standard, Lightweight 모드로 축약한다.

첫 분석 대상 후보:

- 요구사항정의서
- 테스트 계획서
- 테스트 결과서
- 요구사항 추적 매트릭스
