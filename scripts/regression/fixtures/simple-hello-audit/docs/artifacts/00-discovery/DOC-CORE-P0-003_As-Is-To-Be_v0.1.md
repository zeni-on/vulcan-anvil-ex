## 1. 문서 목적

본 문서는 현재 업무/시스템의 상태와 개선 후 목표 상태를 비교해 요구사항 도출의 근거를 만든다.

As-Is/To-Be 정의는 기능명세와 프로그램 설계가 문제 해결 방향에서 벗어나지 않도록 하는 기준으로 사용한다.

## 2. 작성 기준

- 현재 상태는 사실과 관찰 근거를 중심으로 작성한다.
- 개선 방향은 구현 방법보다 업무 결과와 사용자 경험 변화를 우선한다.
- 데이터, 보안, 운영, 성능 관점의 차이를 함께 기록한다.
- 개선 방향에서 요구사항으로 내려갈 항목은 `REQ 후보`로 표시한다.

## 3. 현행 요약

| 항목 | 내용 |
| --- | --- |
| 현행 업무/시스템 | 실제 백엔드 애플리케이션 소스가 아직 없다. |
| 주요 사용자 | 프로세스 검증자, 개발자 |
| 주요 데이터 | 없음. hello 응답만 사용한다. |
| 주요 제약 | Phase 0이므로 구현 전에 요구사항, 설계, 테스트 기준 승인이 필요하다. |
| 주요 문제 | Gate/Run/추적성 흐름이 실제 작은 기능 구현으로 검증되지 않았다. |
