## 1. 문서 목적

본 문서는 프로젝트의 기능 요구사항, 비기능 요구사항, 인수기준, 범위, 제약사항을 정의한다.

요구사항정의서는 Gate 1의 핵심 산출물이며, 이후 기능명세서, 화면설계서, 프로그램 설계서, DB명세서, 테스트계획, 요구사항추적표의 기준이 된다.

## 2. 작성 기준

- 요구사항은 구현 방법이 아니라 사용자가 기대하는 결과와 시스템이 제공해야 할 능력을 설명한다.
- 기능 요구사항은 `REQ-NNN` 형식으로 부여한다.
- 상세 기능 요구사항은 필요 시 `REQ-NNN-NN` 형식으로 부여한다.
- 비기능 요구사항은 `NREQ-NNN` 형식으로 부여한다.
- 각 요구사항은 하나 이상의 인수기준 `AC-NNN-NN`과 연결하는 것을 원칙으로 한다.
- 외부 RFP, 감리, 고객 문서 ID가 있으면 `외부 ID`에 보존한다.
- 보안 요구사항은 적용 가능한 `SEC-ID`와 외부 보안 기준 `SR-ID`를 함께 기록한다.
- 데이터 항목이 포함된 요구사항은 공공데이터 공통표준 용어/도메인 검토 여부를 기록한다.
- 불명확한 요구사항은 추정하지 말고 `확인 필요`로 표시한다.
- Gate 1에서는 구현 파일, 테스트 코드, 화면 시안, DB 물리 설계를 작성하지 않는다.
- Gate 1에서 기술스택, 인증 방식, DB 제품 같은 구현 의사결정이 필요하면 요구사항 본문이 아니라 `DEC-ID`로 분리한다.

## 7. 비기능 요구사항

| NREQ-ID | 구분 | 요구사항명 | 설명 | 측정/검증 기준 | 참조 표준 | 우선순위 | 상태 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| NREQ-001 | 운영 | 로컬 재현성 | 개발자 또는 검증자가 로컬 환경에서 백엔드를 실행하고 API 응답을 검증할 수 있어야 한다. | 실행 명령과 검증 명령이 Gate 2/3 산출물에 정의되고 Gate 4에서 실행 결과가 남아야 한다. | TECH_STACK_BASELINES | Must | Draft |

## 8. 인수기준

인수기준은 요구사항이 충족되었는지 판단할 수 있는 검증 가능한 문장이어야 한다.
요구사항정의서에서는 인수기준을 표로만 관리한다. `Given/When/Then`, 선행조건, 예외조건, 검증 데이터, 테스트 절차는 Gate 3 테스트케이스 문서에 작성한다.

### 8.1 상세 요구사항과 인수기준 매핑

| 상세 REQ-ID | 상세 요구사항명 | 설명 | 관련 AC-ID | 상태 |
| --- | --- | --- | --- | --- |
| REQ-001-01 | hello 응답 반환 | API 호출 시 성공 상태와 `hello` 응답을 반환한다. | AC-001-01, AC-001-02 | Draft |
| NREQ-001 | 로컬 재현성 | 로컬 실행과 검증 명령이 재현 가능해야 한다. | AC-002-01 | Draft |

### 8.2 인수기준 목록

| AC-ID | 관련 요구사항 | 인수기준 | 검증 방식 | 상태 |
| --- | --- | --- | --- | --- |
| AC-001-01 | REQ-001-01 | hello API 호출 결과가 성공 상태로 반환된다. | 통합테스트 / 명령 기반 검증 | Draft |
| AC-001-02 | REQ-001-01 | hello API 응답 본문은 단순 문자열 `hello`와 일치한다. | 통합테스트 / 명령 기반 검증 | Draft |
| AC-002-01 | NREQ-001 | 로컬 실행 명령과 검증 명령이 문서화되고 동일 명령으로 재현 가능하다. | 리뷰 / 명령 기반 검증 | Draft |

작성 기준:

- `관련 요구사항`은 가능한 한 `REQ-NNN-NN` 상세 요구사항으로 연결한다.
- 단위테스트가 필요 없고 통합테스트 또는 UI 검증으로 충분하면 `검증 방식`에 그 이유를 짧게 남긴다.
- `미정`, `확인 필요`가 남아 있으면 Gate 1 완료 전 검토 항목으로 올린다.
